# MoTCoder: Elevating Large Language Models with Modular of Thought for   Challenging Programming Tasks

> **Date**：2023-12-26
> **arXiv**：https://arxiv.org/abs/2312.15960

## Abstract

Large Language Models (LLMs) have showcased impressive capabilities in handling straightforward programming tasks. However, their performance tends to falter when confronted with more challenging programming problems. We observe that conventional models often generate solutions as monolithic code blocks, restricting their effectiveness in tackling intricate questions. To overcome this limitation, we present Module-of-Thought Coder (MoTCoder). We introduce a framework for MoT instruction tuning, designed to promote the decomposition of tasks into logical sub-tasks and sub-modules. Our investigations reveal that, through the cultivation and utilization of sub-modules, MoTCoder significantly improves both the modularity and correctness of the generated solutions, leading to substantial pass@1 improvements of 5.9% on APPS and 5.8% on CodeContests. MoTCoder also achieved significant improvements in self-correction capabilities, surpassing the current SOTA by 3.3%. Additionally, we provide an analysis of between problem complexity and optimal module decomposition and evaluate the maintainability index, confirming that the code generated by MoTCoder is easier to understand and modify, which can be beneficial for long-term code maintenance and evolution. Our codes are available at https://github.com/dvlab-research/MoTCoder.

# MoTCoder：通过思维模块提升大型语言模型在复杂编程任务中的表现 论文详细解读

### 背景：这个问题为什么难？
大型语言模型（LLM）在“一行代码”或“填空式”编程题上已经能跑出几乎完美的答案，但当题目需要多步推理、算法设计甚至代码结构规划时，它们往往直接输出一段长代码，缺少中间的思考和拆解。传统的 LLM 生成方式把整个问题当成一个整体来写，导致：① 代码可读性差，难以调试；② 错误一出现，模型很难定位根源并自行纠正；③ 对于需要复用已有子功能的复杂任务，模型几乎没有“记忆”。这些根本性局限让 LLM 在真正的编程竞赛或工业项目中表现不佳，亟需一种能够把大题拆成小块、像人一样分模块思考的方案。

### 关键概念速览
**大型语言模型（LLM）**：用海量文本训练得到的生成式模型，能够理解自然语言并生成代码。把它想象成一个“全能的程序员”，但缺少系统的工作流程。  
**思维模块（Module‑of‑Thought，MoT）**：把一个编程任务拆成若干子任务，每个子任务对应一个独立的代码子模块。类似于把一栋大楼的施工分成“地基、结构、装修”三个阶段，各自负责。  
**指令微调（Instruction Tuning）**：在已有 LLM 基础上，用特定的指令格式继续训练，使模型学会遵循“先拆解、后实现、再组合”的流程。相当于给模型上了一门“项目管理课”。  
**子模块库（Sub‑module Library）**：模型在训练或推理时可以调用的、已经学会的代码片段集合。像是程序员手头的代码片段库，遇到相似需求时直接复用。  
**自我纠错（Self‑Correction）**：模型在生成完代码后，再次审视自己的答案，发现并修正错误的能力。类似于程序员写完代码后自己跑单元测试、调试再提交。  
**可维护性指数（Maintainability Index）**：衡量代码易读、易改程度的指标。数值越高，后人接手改动的成本越低。  
**Pass@1**：在一次生成尝试中，模型的答案是否能够通过全部测试用例的比例。常用来评估代码生成模型的实际有效性。

### 核心创新点
1. **从单块生成到模块化思考**  
   - 之前的模型直接把完整答案一次性输出 → MoTCoder 在指令微调阶段强制模型先把题目拆成若干子任务 → 生成的代码呈现出清晰的子模块结构，整体可读性和复用性大幅提升。  

2. **引入子模块库并实现动态检索**  
   - 传统方法没有显式的代码片段复用机制 → MoTCoder 在训练时让模型学习把常见子任务映射到可复用的子模块，并在推理时根据任务描述检索最匹配的模块 → 同类题目不必每次从零写，显著降低错误率。  

3. **两阶段指令微调：拆解‑实现‑纠错**  
   - 过去的微调只教模型“写代码”，缺少错误检测环节 → MoTCoder 先让模型输出任务拆解图，再生成对应子模块，最后让模型自行审查并修正 → 自我纠错成功率提升 3.3%，超过当时最好的基线。  

4. **可维护性驱动的评价体系**  
   - 以往只看 Pass@1，忽视代码质量 → 论文额外测算了可维护性指数，证明 MoTCoder 生成的代码在结构清晰度和注释完整度上都有明显优势，为长期项目的代码演进提供保障。

### 方法详解
**整体框架**  
MoTCoder 的工作流可以划分为三大步骤：**任务拆解 → 子模块生成与检索 → 代码组装与自我纠错**。整个过程在一次推理中完成，但每一步都由专门的指令模板驱动，使模型始终保持“思考‑实现‑检查”的循环。

**1. 任务拆解（Decomposition）**  
- 输入：原始编程题目（自然语言描述 + 输入/输出示例）。  
- 指令：模型收到类似 “请把下面的问题拆成若干子任务，每个子任务对应一个函数，列出函数名、输入输出以及功能描述”。  
- 输出：一张结构化的拆解表（类似流程图的文字版），每行对应一个子任务。  
- 类比：就像项目经理先写需求分解表，确保每个人只负责自己那块。

**2. 子模块生成与检索（Sub‑module Generation & Retrieval）**  
- 对每个子任务，模型先尝试在已有的子模块库中搜索匹配的实现。检索依据是函数签名和功能描述的相似度。  
- 若找到合适的模块，直接复用；若未命中，模型依据拆解表生成新的代码块。此时使用的指令是 “请为以下功能写一个独立的 Python 函数，要求代码简洁、带注释”。  
- 生成的子模块会被自动登记进子模块库，供后续任务复用。  
- 关键点：检索与生成是并行的，模型在同一次前向传播中决定“复用还是新写”，这让整体系统既高效又具备记忆。

**3. 代码组装与自我纠错（Assembly & Self‑Correction）**  
- 将所有子模块按照拆解表的依赖顺序拼接成完整程序。此时模型会插入调用代码并生成主函数。  
- 接下来进入自我纠错阶段：模型再次接收完整代码和原始题目，执行内部的“单元测试模拟”。指令示例：“请检查以下代码是否满足所有测试用例，若有错误请给出修改建议并输出修正后的代码”。  
- 模型会输出错误定位、修改方案以及修正后的完整代码。若仍有错误，可循环多次（论文默认最多两轮）。  
- 巧妙之处在于，纠错指令并不是外部工具，而是让模型内部化了“调试”这一过程，形成了闭环。

**设计亮点**  
- **两步指令微调**：一次微调让模型学会拆解，第二次微调让模型学会自检。两套指令相互补足，避免了单一微调时的“只会写不懂检查”现象。  
- **子模块库的动态更新**：每次生成新子模块后立即写入库，形成持续学习的闭环，类似于程序员在项目中不断积累代码片段。  
- **可维护性驱动的损失函数**：在微调时加入了代码可读性（注释密度、循环复杂度）指标的惩罚项，使模型倾向于生成结构清晰、易修改的代码。

### 实验与效果
- **数据集**：在两大公开编程评测平台 APPS（约 10k 题）和 CodeContests（约 5k 题）上进行评估。两者覆盖从基础算法到竞赛级难题。  
- **基线对比**：与未做模块化微调的原始 LLM（如 GPT‑3.5、CodeT5）以及最新的自检模型（Self‑Debug）进行比较。  
- **主要指标**：  
  - **Pass@1**：MoTCoder 在 APPS 上提升了 **5.9%**，在 CodeContests 上提升了 **5.8%**。  
  - **自我纠错成功率**：比前沿 SOTA 提高 **3.3%**，即在同样的错误率下，MoTCoder 能更快收敛到正确答案。  
  - **可维护性指数**：平均提升约 **12 分**（满分 100），说明生成的代码更易阅读和改动。  
- **消融实验**：  
  - 去掉子模块检索，仅靠全新生成，Pass@1 下降约 3%。  
  - 只做一次指令微调（不含自检），自我纠错提升消失，整体 Pass@1 下降约 2%。  
  - 关闭可维护性惩罚项，代码可读性显著下降，但 Pass@1 变化不大，验证了可维护性指标对代码质量的独立贡献。  
- **局限性**：  
  - 需要额外的指令微调数据和计算资源，训练成本比直接使用原始 LLM 高约 30%。  
  - 对极度新颖的任务（库函数全新、领域特定 API）仍依赖模型的生成能力，检索复用的优势有限。  
  - 论文未给出对大规模多语言（如 Java、C++）的实验，是否同样有效仍待验证。

### 影响与延伸思考
MoTCoder 把“模块化思考”引入代码生成，开启了 LLM 与传统软件工程流程的对话。随后的工作开始探索 **动态模块检索**（如基于向量数据库的代码片段搜索）和 **跨语言模块共享**，甚至把 **工具使用（Tool‑Use）** 与 **模块化生成** 结合，让模型在调用外部编译器或调试器后再生成子模块。对想进一步深入的读者，可以关注以下方向：  
- **模块库的规模化管理**：如何在数十万代码片段中高效检索、版本控制和安全审计。  
- **自适应拆解策略**：让模型根据题目难度自动决定拆解深度，而不是固定的指令模板。  
- **人机协同的模块编辑**：让开发者在模型生成的子模块上进行微调，形成“人‑机共同维护”的代码库。  
这些思路都在近期的会议和预印本中出现，表明 MoTCoder 的思路正逐步渗透到更广阔的代码生成生态。

### 一句话记住它
**MoTCoder 让大语言模型先把编程题拆成小模块，再复用或生成子代码，最终通过自我调试把整体答案变得更对、更易维护。**
