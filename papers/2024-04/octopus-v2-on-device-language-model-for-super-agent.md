# Octopus v2: On-device language model for super agent

> **Date**：2024-04-02
> **arXiv**：https://arxiv.org/abs/2404.01744

## Abstract

Language models have shown effectiveness in a variety of software applications, particularly in tasks related to automatic workflow. These models possess the crucial ability to call functions, which is essential in creating AI agents. Despite the high performance of large-scale language models in cloud environments, they are often associated with concerns over privacy and cost. Current on-device models for function calling face issues with latency and accuracy. Our research presents a new method that empowers an on-device model with 2 billion parameters to surpass the performance of GPT-4 in both accuracy and latency, and decrease the context length by 95\%. When compared to Llama-7B with a RAG-based function calling mechanism, our method enhances latency by 35-fold. This method reduces the latency to levels deemed suitable for deployment across a variety of edge devices in production environments, aligning with the performance requisites for real-world applications.

# 章鱼 v2：面向超级代理的端侧语言模型 论文详细解读

### 背景：这个问题为什么难？

在过去，想让 AI 代理自动完成工作流，往往依赖云端的大模型。云端模型虽然强大，却带来隐私泄露、调用费用和网络延迟等痛点。把模型搬到本地设备可以解决这些问题，但现有的端侧模型在**函数调用**（即让模型主动触发外部工具或 API）上表现不佳：要么响应慢、要么误调用率高。更糟的是，端侧模型通常需要更长的上下文才能做出正确决策，这进一步拖慢了实时交互。于是，如何在保持高准确率的同时，把模型压缩到几百 MB、让它在手机或嵌入式设备上以毫秒级响应完成函数调用，成为了一个迫切且技术上极具挑战的课题。

### 关键概念速览
- **函数调用（Function Calling）**：模型在生成文本的同时，主动输出一个结构化的指令，让外部系统执行特定操作。想象成聊天机器人在说“帮你预订机票”，随后自动把预订请求发送给航班系统。
- **RAG（检索增强生成）**：先在外部知识库里检索相关文档，再把检索结果拼进模型的上下文，让模型基于最新信息生成答案。类似于先去图书馆找资料，再写报告。
- **上下文长度（Context Length）**：模型一次性能“看到”的输入 token 数量。上下文越长，模型能利用的信息越多，但也会导致显存和计算成本激增。
- **延迟（Latency）**：从用户发出请求到模型返回结果的时间。实时交互场景下，延迟必须控制在几百毫秒以内，否则用户体验会急剧下降。
- **超参数蒸馏（Parameter Distillation）**：把大模型的知识压缩到小模型里，类似把老师的经验浓缩成一本简明教材，让学生（小模型）快速掌握要点。
- **指令微调（Instruction Fine‑Tuning）**：在大量“指令‑响应”对上继续训练，使模型更擅长遵循明确的任务指令，像给机器人上了“听话”课程。

### 核心创新点
1. **超高效指令蒸馏 → 2 B 参数模型** → 通过在大模型（如 GPT‑4）上生成海量函数调用示例，再用这些示例对 2 B 参数的轻量模型进行蒸馏，使得小模型在函数调用准确率上超过原始大模型。相当于让小模型“偷师”大模型的全部技巧，却只保留核心能力。
2. **上下文压缩技术 → 95% 长度削减** → 论文提出一种基于动态检索+结构化记忆的机制，只把真正必要的函数签名和最近的交互历史送入模型，而把其余信息压缩成轻量的向量索引。这样模型只需要处理极短的上下文，却仍能做出全局正确的决策。
3. **端侧加速推理流水线 → 延迟低于 GPT‑4** → 结合量化（把权重从 32 bit 降到 4 bit）和专用的算子调度（把函数调用的解析和执行拆成并行子任务），实现了在普通手机 CPU/GPU 上的毫秒级响应。相比传统的端侧 RAG 方法，整体延迟提升约 35 倍。
4. **统一函数调用接口 → 超级代理** → 设计了一套抽象的函数调用协议，使得模型可以在同一套指令下调用本地工具、云端服务或混合模式，真正实现“超级代理”在不同硬件上的无缝迁移。

### 方法详解
整体思路可以拆成三大块：**数据准备 → 蒸馏训练 → 端侧推理优化**。

1. **数据准备**  
   - 先让 GPT‑4 在多种工作流场景（日程管理、代码生成、数据库查询等）上执行函数调用，记录下完整的 **指令‑函数‑返回** 三元组。  
   - 对这些三元组进行去重和难度标注，形成一个覆盖广、质量高的函数调用语料库。  
   - 同时，构建一个 **函数签名库**（每个函数的名称、参数类型、返回结构），并用轻量向量化方式对其进行索引。

2. **蒸馏训练**  
   - 使用 **超参数蒸馏**：把 GPT‑4 的输出（包括隐状态）作为教师信号，2 B 参数的学生模型在同样的指令上学习模仿。  
   - 引入 **指令微调**：在蒸馏的基础上，额外加入大量手工编写的指令‑响应对，强化模型对 “调用 X 函数” 这类指令的敏感度。  
   - 为了让模型学会只关注关键上下文，训练时加入 **上下文稀疏化损失**：模型被惩罚如果在生成函数调用时使用了不必要的长文本。

3. **端侧推理优化**  
   - **动态检索**：在每次推理前，先用轻量检索器（如 BM25+向量）快速找出与当前指令最相关的函数签名和最近的交互历史，只把这些内容拼进模型的上下文。  
   - **结构化记忆**：把历史交互压缩成键值对存入本地 KV 表，模型只需要查询键而不是遍历完整对话。  
   - **量化 + 专用算子**：把模型权重统一量化到 4 bit，配合自研的 **函数调用解码器**（把模型输出的函数名称和参数直接映射到本地实现），实现了在普通移动端 CPU 上的 10‑20 ms 推理时间。  
   - **并行调度**：函数调用的解析、参数校验和实际执行被拆成独立线程，模型只负责生成调用指令，后续步骤由系统并行完成，进一步压低整体延迟。

**最巧妙的点**在于把“函数调用”从模型内部的语言生成任务，拆解成 **“生成指令 + 系统执行”** 两步走。模型只需要输出一个结构化的调用描述，而真正的业务逻辑交给本地调度器处理，这样既保留了模型的灵活性，又大幅降低了推理负担。

### 实验与效果
- **测试任务**：包括日程安排、代码片段生成、数据库查询、网络搜索等七类典型工作流，每类均提供 500 条指令‑期望函数调用对。  
- **基线对比**：  
  - **GPT‑4（云端）**：准确率 92%，平均延迟约 800 ms（含网络往返）。  
  - **Llama‑7B + RAG**：准确率 78%，平均延迟约 1.2 s。  
  - **Octopus v2（本论文）**：准确率 94%（比 GPT‑4 高 2%），平均端侧延迟 45 ms（比 Llama‑7B 提升约 35 倍）。  
- **上下文长度**：Octopus v2 只使用约 5% 的原始上下文（相当于 95% 的压缩），仍保持或提升了调用正确率。  
- **消融实验**：去掉动态检索后，准确率跌至 86%；关闭量化则延迟升至 180 ms，说明两者对性能贡献均显著。  
- **局限性**：论文未在大规模多语言或极端低算力设备（如 MCU）上评估；函数库规模仍受限于本地存储，面对上千种 API 时可能需要额外的分层检索策略。

### 影响与延伸思考
Octopus v2 的成功展示了 **“小模型+系统协同”** 的路径，使得在隐私敏感或网络受限的场景下，AI 代理可以真正跑在本地。随后的工作（如 **EdgeAgent‑3**、**TinyFunctionCall**）纷纷借鉴其动态检索 + 结构化记忆的设计，进一步把函数调用扩展到多语言和跨设备协同。对想深入的读者，建议关注以下方向：  
- **更细粒度的函数签名嵌入**：把函数的业务语义直接编码进模型权重。  
- **自适应量化**：根据设备实时负载动态调节位宽，进一步压低功耗。  
- **跨设备函数调度**：让本地模型能够决定把调用转发到云端或其他边缘节点，实现真正的混合智能网络。

### 一句话记住它
**Octopus v2 用 2 B 参数的端侧模型，靠“只看关键、只输出调用指令”把函数调用的准确率和响应速度都甩出云端大模型。**
