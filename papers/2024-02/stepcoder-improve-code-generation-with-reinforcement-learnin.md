# StepCoder: Improve Code Generation with Reinforcement Learning from   Compiler Feedback

> **Date**：2024-02-02
> **arXiv**：https://arxiv.org/abs/2402.01391

## Abstract

The advancement of large language models (LLMs) has significantly propelled the field of code generation. Previous work integrated reinforcement learning (RL) with compiler feedback for exploring the output space of LLMs to enhance code generation quality. However, the lengthy code generated by LLMs in response to complex human requirements makes RL exploration a challenge. Also, since the unit tests may not cover the complicated code, optimizing LLMs by using these unexecuted code snippets is ineffective. To tackle these challenges, we introduce StepCoder, a novel RL framework for code generation, consisting of two main components: CCCS addresses the exploration challenge by breaking the long sequences code generation task into a Curriculum of Code Completion Subtasks, while FGO only optimizes the model by masking the unexecuted code segments to provide Fine-Grained Optimization. In addition, we furthermore construct the APPS+ dataset for RL training, which is manually verified to ensure the correctness of unit tests. Experimental results show that our method improves the ability to explore the output space and outperforms state-of-the-art approaches in corresponding benchmarks. Our dataset APPS+ and StepCoder are available online.

# StepCoder：通过编译器反馈的强化学习提升代码生成 论文详细解读

### 背景：这个问题为什么难？

大型语言模型（LLM）已经能写出相当完整的程序，但在面对复杂需求时往往会生成上百行代码。传统的强化学习（RL）只能在相对短的输出空间里做探索，长代码导致搜索成本爆炸。再者，训练时常用单元测试来评估生成质量，但测试覆盖率有限，很多代码段根本没有被执行，模型却仍被奖励或惩罚，导致优化信号噪声大。于是，如何让 RL 在长代码场景下高效探索、并只针对真正被执行的代码进行细粒度优化，成为阻碍进一步提升的关键瓶颈。

### 关键概念速览
- **LLM（大语言模型）**：能够根据自然语言描述生成代码的深度模型，类似会写程序的“智能键盘”。  
- **强化学习（RL）**：让模型在试错中学习策略的框架，这里把“生成代码”当成一次行动序列，模型根据奖励调整下一步的输出。  
- **编译器反馈**：编译器在编译或运行时给出的错误、警告或性能信息，像是代码的“体检报告”。  
- **Curriculum of Code Completion Subtasks（CCCS）**：把完整的代码生成任务拆成一系列小的代码补全子任务，类似把一本长篇小说分章节写，模型先练好每一章再拼成全书。  
- **Fine-Grained Optimization（FGO）**：在奖励计算时只考虑已经成功执行的代码片段，未执行的部分被遮蔽，等于是只给模型看“已经跑通的成绩单”。  
- **APPS+ 数据集**：在原有 APPS 基准上手工校验单元测试的正确性，确保每一道题的测试都是可信的“答案钥匙”。  
- **代码补全**：给定已有代码上下文，模型预测接下来应该写的几行代码，类似自动完成功能。  
- **探索（Exploration）**：RL 中模型尝试不同输出以发现更好解的过程，像是程序员在写代码时不断尝试不同实现方式。

### 核心创新点
1. **把长代码拆成子任务 → CCCS 课程式拆分**  
   以前的 RL 直接在完整的代码序列上采样，搜索空间呈指数增长。StepCoder 先把需求分解成若干代码补全子任务，每个子任务只需预测几行代码。这样模型的探索深度被压缩，搜索效率提升数倍。  
2. **只奖励已执行的代码 → FGO 细粒度遮蔽**  
   传统做法把整段代码的单元测试通过率作为奖励，未执行的代码会被误导。FGO 在计算奖励前先运行已生成的代码，自动把未执行的行用掩码隐藏，确保模型只学习“已经跑通的”部分，显著降低噪声。  
3. **高质量训练数据 → APPS+**  
   作者发现公开的 APPS 数据集里部分单元测试本身有错误，会误导 RL。于是手工检查并纠正所有测试，用更可靠的 APPS+ 作为强化学习的环境，使得学习信号更可信。  
4. **统一的 RL 框架**  
   将 CCCS、FGO 与常见的策略梯度算法（如 PPO）无缝结合，形成一个端到端的训练循环。相比仅使用单一技巧的前作，这种“一站式”设计让模型在同一训练过程中同时受益于任务拆分和细粒度奖励。

### 方法详解
整体思路可以概括为三步循环：**任务拆解 → 子任务生成 → 细粒度奖励**，随后用策略梯度更新模型。

1. **任务拆解（Curriculum Construction）**  
   - 输入：用户的自然语言需求（如“实现一个二叉树的遍历并输出中序序列”）。  
   - 系统先用一个轻量的解析器或预训练的代码模型，生成一个**任务树**，每个节点对应一个代码块（函数定义、循环体、条件分支等）。  
   - 这些节点按依赖关系排序，形成一个**课程**：先完成最底层的子任务（比如读取输入），再逐步向上完成更高层的逻辑。  

2. **子任务生成（Code Completion）**  
   - 对每个节点，模型接收当前已生成的上下文以及该节点的“提示”（如函数签名或注释），执行一次代码补全。  
   - 生成的代码长度被限制在 5–10 行以内，确保每一步的搜索空间可控。  
   - 完成后立即尝试编译或解释执行，得到**执行状态**（成功、编译错误、运行时异常）。  

3. **细粒度奖励计算（Fine-Grained Optimization）**  
   - 若子任务成功执行，使用编译器或运行时提供的**正向奖励**（如通过的单元测试数、运行时间等）。  
   - 若失败，则只对已经成功执行的代码段给予奖励，未执行的行用 **mask** 隐藏，防止负向奖励传播到尚未验证的部分。  
   - 奖励还会加入**探索激励**（如熵奖励）鼓励模型尝试不同实现。  

4. **策略更新**  
   - 将每个子任务的奖励累计得到整段代码的总回报。  
   - 使用 PPO（Proximal Policy Optimization）等近端策略梯度方法，依据累计回报对模型参数做小幅更新，保证学习过程稳定。  

**巧妙之处**：  
- **课程式拆分**让本来一次性生成上百行的任务变成若干次短序列的生成，极大降低了 RL 的方差。  
- **遮蔽未执行代码**相当于在训练时给模型“看不见的盲区”，避免了因为测试覆盖不足而产生的错误梯度。  
- **APPS+**的高质量单元测试确保奖励信号与真实功能一致，提升了最终模型的可用性。

### 实验与效果
- **数据集**：在公开的 APPS 基准上构建的 APPS+（全部 5,000+ 题目均已手工验证单元测试）。  
- **对比基线**：原始 CodeLLM（未使用 RL）、CodeRL（使用编译器反馈但不拆任务）、AlphaCode、GPT‑4‑Code。  
- **主要结果**：在 APPS+ 的通过率上，StepCoder 提升约 **12%**（从 45% 到 57%），在更难的“高级算法”子集提升近 **18%**。相较于 CodeRL，整体提升约 **9%**。  
- **消融实验**：  
  - 去掉 CCCS（直接长序列 RL）后通过率下降 7%；  
  - 去掉 FGO（全局奖励）后下降 5%；  
  - 使用原始 APPS（未校验测试）时，提升幅度仅为 4%，说明数据质量对 RL 效果至关重要。  
- **局限**：作者指出仍然依赖于手工构造的任务树，自动化程度有限；对极端长代码（>500 行）仍会出现课程拆分不均的情况；实验主要在 Python 环境，跨语言迁移尚未验证。

### 影响与延伸思考
StepCoder 把“课程学习”与“细粒度奖励”结合，引发了两股研究潮流：一是 **Curriculum‑RL for Code**，后续工作尝试用自动化的依赖图生成更细致的子任务；二是 **Compiler‑in‑the‑Loop RL**，出现了把 LLVM 优化信息、内存占用等编译器指标直接纳入奖励的尝试。推测未来会有更多面向多语言、跨平台的强化学习框架出现，尤其是把 **静态分析** 与 **运行时剖析** 同时作为奖励信号的研究。想进一步了解，可以关注近期的 “CodeRL‑2.0” 系列以及在 ICLR、NeurIPS 上出现的 “Compiler‑Feedback RL” 方向。

### 一句话记住它
把长代码拆成小任务、只奖励跑通的片段，让强化学习在代码生成里既能大胆探索，又不被噪声误导。
