# $R^3$: "This is My SQL, Are You With Me?" A Consensus-Based Multi-Agent   System for Text-to-SQL Tasks

> **Date**：2024-02-20
> **arXiv**：https://arxiv.org/abs/2402.14851

## Abstract

Large Language Models (LLMs) have demonstrated strong performance on various tasks. To unleash their power on the Text-to-SQL task, we propose $R^3$ (Review-Rebuttal-Revision), a consensus-based multi-agent system for Text-to-SQL tasks. $R^3$ outperforms the existing single LLM Text-to-SQL systems as well as the multi-agent Text-to-SQL systems by $1.3\%$ to $8.1\%$ on Spider and Bird. Surprisingly, we find that for Llama-3-8B, $R^3$ outperforms chain-of-thought prompting by over 20\%, even outperforming GPT-3.5 on the development set of Spider.

# R³：“这是我的SQL，你跟上吗？”——基于共识的多智能体文本到SQL系统 论文详细解读

### 背景：这个问题为什么难？
把自然语言问题直接翻译成结构化的SQL查询，看似只要让模型学会对应关系，却在实际中碰到很多障碍。第一，SQL语法严格，稍有语序或关键字错误就会导致执行失败。第二，用户提问往往隐含多步推理——先要识别涉及的表、列，再要决定连接方式、过滤条件等，这对一次性生成的模型来说容易遗漏。第三，单一的大语言模型（LLM）虽然能生成合理的SQL，但缺乏自我纠错和多视角审视的机制，导致在复杂查询上的准确率停滞不前。正是这些根本性限制，让研究者们开始探索多智能体协作的思路。

### 关键概念速览
**文本到SQL（Text-to-SQL）**：把自然语言问句自动转化为对应的SQL查询语句，类似把口头指令翻译成数据库指令。  
**大语言模型（LLM）**：拥有上百亿参数的预训练模型，能够理解并生成自然语言文本，例如GPT‑3.5、Llama‑3。  
**Chain‑of‑Thought（思维链）**：让模型在给出答案前先写出推理步骤，就像解题时先列出思路，帮助提升复杂任务的准确率。  
**多智能体系统**：由多个相互独立的模型（或同一模型的不同实例）组成，每个负责特定子任务，最终通过协商达成统一答案。  
**共识机制（Consensus）**：所有智能体对同一问题给出答案后，通过投票或加权平均等方式决定最终输出，类似团队讨论后达成的共识。  
**Review‑Rebuttal‑Revision（审稿‑反驳‑修订）**：本论文提出的三轮交互流程，先让模型生成初稿（Review），再让其他模型指出错误（Rebuttal），最后根据反馈进行修改（Revision）。  
**Spider / BIRD 数据集**：两个业界常用的 Text‑to‑SQL 基准，包含上百个跨表复杂查询，用来衡量模型的实际能力。

### 核心创新点
**单轮生成 → 三轮审稿机制 → 准确率大幅提升**  
传统方法让 LLM 直接一次性输出 SQL，错误难以纠正。R³ 把生成过程拆成 Review、Rebuttal、Revision 三步：第一轮模型自行生成初稿；第二轮其他模型专门检查语法、表结构匹配等潜在错误并给出批评；第三轮模型根据批评重新生成或修正。这样相当于让模型“自我审稿”，显著降低了低级错误。

**单模型 → 多智能体共识 → 稳定性增强**  
以前的多智能体方案往往让每个模型独立输出，然后随意选取一个。R³ 引入共识层，所有智能体的输出经过投票或加权融合，只有多数同意的部分才进入下一轮。这样可以抑制单个模型的偶然失误，使最终 SQL 更可靠。

**通用提示 → 专用角色提示 → 任务分解更细**  
R³ 为每个阶段设计了专属的系统提示（system prompt），让模型在 Review 时专注于“写出完整查询”，在 Rebuttal 时专注于“找出潜在错误”，在 Revision 时专注于“根据批评改进”。这种角色化提示把同一模型的能力拆分成不同“职业”，提升了每一步的专注度。

**轻量级协作 → 超越链式思维**  
链式思维（CoT）通过让模型一步步思考提升表现，但仍是单体内部的推理。R³ 把推理外包给其他模型，让它们相互“辩论”。实验显示，即使是 8 B 参数的 Llama‑3，在这种协作下也能比 CoT 提升 20% 以上，甚至超过更大的 GPT‑3.5。

### 方法详解
整体框架可以想象成一次三轮的团队会议。首先，**Review** 阶段启动一个“作者”智能体，它接收用户的自然语言问题和数据库 schema（表结构）信息，生成一条完整的 SQL 初稿。此时模型的系统提示会强调“写出可直接执行的查询”。  
接下来进入 **Rebuttal** 环节，系统会召集若干“审稿人”智能体（数量可调），每个审稿人拿到同样的输入和 Review 的输出，但提示被改成“检查以下 SQL 是否符合 schema、是否有语法错误、是否遗漏关键条件”。审稿人会返回一段文字，列出发现的问题或直接给出错误位置的标记。  
最后是 **Revision** 阶段，原作者智能体再次被激活，这次的系统提示是“根据审稿人的批评修改 SQL”。它会把 Review 的原始 SQL 与审稿人的反馈一起喂入模型，让模型在保持原意的前提下修正错误。若有多个审稿人给出不同意见，系统会先进行 **共识层**：统计每条批评出现的频率，只有被多数审稿人指出的错误才进入 Revision，避免因单个审稿人的误判导致不必要的改动。  
整个流程可以用文字版流程图表示：

1. 用户问句 + schema → Review 智能体 → 初稿 SQL  
2. 初稿 SQL + schema → 多个 Rebuttal 智能体 → 批评列表  
3. 批评列表 → 共识过滤 → Revision 智能体 → 修订后 SQL → 输出  

值得注意的巧思在于 **角色化提示**：同一个底层模型在不同阶段被包装成不同“职业”，而不是让它一次性兼顾所有任务。这样模型的注意力更集中，错误率自然下降。另一个反直觉点是 **让模型互相批评**，人们常以为模型之间缺乏可信度，但实验表明，多模型的负向反馈比单模型的自我检查更有效，因为每个模型的随机性会在审稿环节被放大，从而更容易捕捉到细微错误。

### 实验与效果
- **数据集**：在 Spider（跨表复杂查询）和 BIRD（更大规模的真实业务查询）两个公开基准上进行评估。  
- **对比基线**：包括单模型直接生成的方案、使用 Chain‑of‑Thought 的 LLM、以及已有的多智能体 Text‑to‑SQL 系统。  
- **提升幅度**：R³ 在 Spider 上比最强单模型提升约 1.3%~8.1%，在 BIRD 上同样取得显著优势。对 Llama‑3‑8B 来说，R³ 的表现比 CoT 提升超过 20%，在 Spider 开发集上甚至超过了 GPT‑3.5。  
- **消融实验**：作者分别去掉 Rebuttal 环节、共识层或角色化提示，发现每一项都对最终准确率有贡献，尤其是去掉共识层后错误率上升约 3%。  
- **局限性**：论文承认 R³ 的推理时间比单模型要长，因为需要多轮交互和多模型调用；在极端超长查询或极度稀疏的 schema 上仍会出现遗漏。  

### 影响与延伸思考
R³ 把“审稿—反驳—修订”引入到 LLM 推理中，开启了“协作式自我纠错”的新思路。随后有几篇工作尝试把类似的三轮机制搬到代码生成、数学证明等任务上，证明这种框架具有跨任务的通用性。对想进一步探索的读者，可以关注以下方向：  
- **高效调度**：如何在保持效果的同时减少模型调用次数，降低成本。  
- **自适应轮数**：根据问题难度动态决定是否需要 Rebuttal 或多轮 Revision。  
- **跨模型共识**：让不同架构（如检索增强模型、专用 SQL 生成模型）共同参与共识，可能进一步提升鲁棒性。  

### 一句话记住它
让大语言模型先写草稿，再让同类模型互相挑刺，最后根据挑刺结果修正——三轮审稿式协作让 Text‑to‑SQL 的准确率飞跃。
