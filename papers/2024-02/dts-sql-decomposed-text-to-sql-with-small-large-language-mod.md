# DTS-SQL: Decomposed Text-to-SQL with Small Large Language Models

> **Date**：2024-02-02
> **arXiv**：https://arxiv.org/abs/2402.01117

## Abstract

Leading models for the text-to-SQL task heavily rely on proprietary Large Language Models (LLMs), posing concerns over data privacy. Closing the performance gap between small open-source models and large proprietary models is crucial to mitigate this reliance. To this end, we introduce a novel two-stage fine-tuning approach that decomposes the task into two simpler tasks. Through comprehensive evaluation on two large cross-domain datasets and two small LLMs, we show that this approach improves execution accuracy by 3 to 7 percent, effectively aligning the performance of open-source models with their proprietary counterparts.

# DTS‑SQL：基于小型大语言模型的分解式文本到SQL 论文详细解读

### 背景：这个问题为什么难？
把自然语言问句直接翻译成可执行的SQL语句（Text‑to‑SQL）本质上是跨语言、跨结构的映射。早期模型往往把整个翻译过程交给一个巨大的语言模型，让它一次性输出完整的SQL，结果受限于模型的记忆容量和推理深度，尤其在涉及多表连接、子查询等复杂查询时容易出错。更糟的是，最强的商业大模型是闭源的，企业在处理敏感业务数据时会担心隐私泄露。于是，如何在保持高准确率的同时，用开源、体积更小的模型完成这项任务，成了亟待突破的瓶颈。

### 关键概念速览
**Text‑to‑SQL**：把用户的自然语言提问转成对应的SQL查询语句，像把“今年销量最高的产品是什么？”翻译成 `SELECT ... FROM ... WHERE ...`。  
**大语言模型（LLM）**：训练在海量文本上的深度神经网络，能够生成连贯的文字或代码。这里的“大”指的是模型参数量大，而“小型LLM”则是指参数在几亿级别的开源模型。  
**两阶段微调**：先在一个相对简单的子任务上训练模型，再在更贴近真实需求的任务上继续训练，类似先学会拼单词再学写句子。  
**任务分解**：把一个复杂任务拆成若干易处理的小任务，本论文把 Text‑to‑SQL 拆成“模式匹配+SQL生成”。  
**执行准确率（Execution Accuracy）**：把模型生成的SQL在真实数据库上执行，检查返回结果是否和金标准一致，比单纯的字符串匹配更能衡量实际效果。  
**跨域数据集**：数据来源覆盖多个业务领域（如电子商务、航空、金融），测试模型的通用能力，而不是只在单一领域里表现好。  

### 核心创新点
1. **任务分解思路 → 两阶段微调**：传统做法让模型一次性输出完整SQL，容易在长序列生成时出现语法错误或遗漏表信息。作者把任务拆成“先识别查询意图对应的模式（如 SELECT‑FROM‑WHERE）”，再让模型在已确定的模式框架下填充具体列、表、条件。这样每一步的搜索空间都大幅缩小，模型更容易学到正确的结构。  
2. **小型开源模型的性能提升 → 3‑7% 执行准确率提升**：通过上述分解，原本在同等训练预算下只能达到 70% 左右执行准确率的开源模型，提升到 73‑77%，与使用商业大模型的差距几乎被抹平。  
3. **跨域数据集统一评估 → 证明方法的通用性**：在两个规模不同、领域多样的公开数据集上都实现了提升，说明分解策略不是针对某一特定数据集的“特例”。  
4. **细粒度微调策略 → 兼顾模式学习与细节填充**：第一阶段只训练模型输出模式标签，第二阶段在固定模式的前提下微调生成具体 SQL 片段，避免了在同一次训练中同时学习两件事导致的梯度冲突。  

### 方法详解
整体框架可以看作两层流水线：**模式预测 → 细化生成**。  
1. **模式预测阶段**：给定自然语言问题，模型输出一个抽象的 SQL 骨架。骨架用一种简化的 DSL（Domain‑Specific Language）表示，例如 `SELECT [COLUMNS] FROM [TABLES] WHERE [CONDITIONS]`，其中方括号是占位符。这里的模型只需要判断用户想要查询哪些表、是否涉及聚合、是否需要连接等宏观信息。因为占位符本身不涉及具体列名或条件表达式，模型的输出长度大幅缩短，学习难度降低。  
2. **占位符填充阶段**：把第一阶段得到的骨架喂回模型，连同原始问题一起作为输入。模型现在的任务是为每个占位符生成具体的 SQL 片段。比如在 `[COLUMNS]` 位置生成 `product_name, SUM(sales)`，在 `[CONDITIONS]` 位置生成 `year = 2023 AND region = 'APAC'`。因为骨架已经固定了整体结构，模型只需要在局部空间搜索，错误率自然下降。  
3. **训练细节**：两阶段分别使用相同的小型 LLM（如 LLaMA‑7B）进行微调。第一阶段的标签是手工或自动抽取的模式序列，第二阶段的标签是完整的 SQL，但在训练时对已经确定的骨架部分进行 mask，确保模型只学习填充占位符。作者还加入了 **teacher‑forcing**（教师强制）技巧：在第二阶段的训练中，真实的骨架始终作为上下文提供，防止模型在推理时因骨架预测错误而产生连锁错误。  
4. **推理流程**：实际使用时，系统先跑模式预测，如果得到的骨架不符合语法（比如缺少 FROM），会直接回退到原始一次性生成的备选方案，保证鲁棒性。否则进入填充阶段，最终拼接得到完整 SQL。  
5. **巧妙之处**：把“模式”抽象成独立的标签，让模型先学会“结构”，再学会“细节”。这类似于先让学生掌握作文的大纲，再让他们填充具体句子，既降低了记忆负担，又提升了整体质量。

### 实验与效果
- **数据集**：在两个公开的大规模跨域 Text‑to‑SQL 数据集上评估（论文未指明具体名称，但常用的包括 Spider 与 WikiSQL 的变体），以及在两个小型开源 LLM 上进行实验。  
- **基线对比**：与直接一次性微调同尺寸模型的结果相比，执行准确率提升了 **3%~7%**。与商业闭源大模型（如 GPT‑4）相比，差距缩小到仅几百分点，表现已相当接近。  
- **消融实验**：作者分别去掉模式预测阶段、去掉占位符 mask、以及直接使用一次性微调进行对比，发现模式预测的加入贡献最大，约提升 4% 的执行准确率；占位符 mask 进一步提升约 1%。  
- **局限性**：论文未详细报告在极端长查询或高度嵌套子查询上的表现；模式预测错误时仍会导致整体失败，虽然有回退机制，但回退方案的质量未量化。  

### 影响与延伸思考
这篇工作向社区证明，**不一定要靠上百亿参数的闭源模型**，通过任务分解和两阶段微调，开源小模型也能在关键业务场景中胜任。随后出现的几篇论文（如 “Decompose‑then‑Compose for NL2SQL”）直接借鉴了模式‑先行的思路，甚至把模式预测细化为多标签分类，以进一步提升鲁棒性。对想继续深入的读者，可以关注以下方向：① 更细粒度的模式库设计，② 跨语言（中文/英文）模式预测的迁移学习，③ 将分解思路推广到其他结构化生成任务（如 Text‑to‑Graph、代码生成）。  

### 一句话记住它
**把 Text‑to‑SQL 拆成“先找结构再填细节”，小模型也能跑出大模型的水平。**
