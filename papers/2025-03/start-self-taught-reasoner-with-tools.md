# START: Self-taught Reasoner with Tools

> **Date**：2025-03-06
> **arXiv**：https://arxiv.org/abs/2503.04625

## Abstract

Large reasoning models (LRMs) like OpenAI-o1 and DeepSeek-R1 have demonstrated remarkable capabilities in complex reasoning tasks through the utilization of long Chain-of-thought (CoT). However, these models often suffer from hallucinations and inefficiencies due to their reliance solely on internal reasoning processes. In this paper, we introduce START (Self-Taught Reasoner with Tools), a novel tool-integrated long CoT reasoning LLM that significantly enhances reasoning capabilities by leveraging external tools. Through code execution, START is capable of performing complex computations, self-checking, exploring diverse methods, and self-debugging, thereby addressing the limitations of LRMs. The core innovation of START lies in its self-learning framework, which comprises two key techniques: 1) Hint-infer: We demonstrate that inserting artificially designed hints (e.g., ``Wait, maybe using Python here is a good idea.'') during the inference process of a LRM effectively stimulates its ability to utilize external tools without the need for any demonstration data. Hint-infer can also serve as a simple and effective sequential test-time scaling method; 2) Hint Rejection Sampling Fine-Tuning (Hint-RFT): Hint-RFT combines Hint-infer and RFT by scoring, filtering, and modifying the reasoning trajectories with tool invocation generated by a LRM via Hint-infer, followed by fine-tuning the LRM. Through this framework, we have fine-tuned the QwQ-32B model to achieve START. On PhD-level science QA (GPQA), competition-level math benchmarks (AMC23, AIME24, AIME25), and the competition-level code benchmark (LiveCodeBench), START achieves accuracy rates of 63.6%, 95.0%, 66.7%, 47.1%, and 47.3%, respectively. It significantly outperforms the base QwQ-32B and achieves performance comparable to the state-of-the-art open-weight model R1-Distill-Qwen-32B and the proprietary model o1-Preview.

# START：自学推理器与工具 论文详细解读

### 背景：这个问题为什么难？

在复杂推理任务里，单纯靠模型内部的“脑子”往往会跑偏——大模型会凭空捏造（hallucination）或在长链式思考（Chain‑of‑Thought）里卡死。过去的做法大多让模型自行完成所有计算，缺少真实的外部算子，导致算术、代码和科学问答的准确率受限。根本的瓶颈在于：模型既要保持通用语言理解，又要拥有可靠的数值/代码执行能力，却没有一个统一的机制把两者结合起来。

### 关键概念速览
- **大推理模型（Large Reasoning Model，LRM）**：参数规模在数十亿以上、专门训练来做多步推理的语言模型。像 OpenAI‑o1、DeepSeek‑R1 属于此类。  
- **思维链（Chain‑of‑Thought，CoT）**：让模型在给出最终答案前把每一步推理写出来，类似人做数学题时的草稿。  
- **工具调用（Tool Use）**：模型在推理过程中主动调用外部程序（如 Python 解释器）来完成计算、检验或生成代码。  
- **Hint‑infer**：在模型生成思维链的过程中，插入人工设计的提示语（如“等一下，可能需要用 Python”），诱导模型主动触发工具调用。可以把它想成在对话里给模型“暗示”去找工具的信号灯。  
- **Hint‑Rejection Sampling Fine‑Tuning（Hint‑RFT）**：先让模型通过 Hint‑infer 产生大量带工具调用的推理轨迹，再用评分、过滤和微调的方式把高质量轨迹固化进模型。相当于先让模型“试错”，再把成功的经验写进记忆。  
- **自检（Self‑checking）**：模型在得到工具输出后再回头检查自己的推理是否前后一致，类似人做完计算后再核对一遍。  
- **自调试（Self‑debugging）**：当工具返回错误或异常时，模型会重新思考、修改提示或代码并再次尝试，像程序员调试代码的循环。

### 核心创新点
1. **Hint‑infer：用提示语驱动工具调用**  
   之前的工具增强模型往往需要大量示例数据来教模型何时使用工具，这成本高且不易迁移。START 直接在推理时插入简短的自然语言提示，模型会自行产生“调用 Python”之类的指令。这样既省去了示例标注，又把工具使用变成了推理的自然一步。  
2. **Hint‑RFT：把高质量的工具调用轨迹写进模型**  
   传统微调只基于已有的标注答案，忽略了推理过程的质量。START 先让模型通过 Hint‑infer 生成大量包含工具调用的完整思维链，然后用评分函数挑出最靠谱的轨迹，再进行有针对性的微调。结果是模型在没有外部提示的情况下也能主动使用工具。  
3. **统一的长链式推理 + 工具执行框架**  
   过去的系统要么只做长 CoT（缺少真实计算），要么只做短的工具调用（缺少全局思考）。START 把两者结合：在长链的每一步都可能触发工具，工具结果再回流到思维链，形成闭环。  
4. **自检 + 自调试的循环机制**  
   当工具返回的数值或代码不符合预期时，模型会自动生成“重新思考”或“改写代码”的新步骤，而不是直接接受错误答案。这大幅降低了 hallucination 的概率。

### 方法详解
**整体框架**  
START 的运行可以分为三大阶段：① 提示驱动的推理生成（Hint‑infer），② 工具执行与结果回馈，③ 基于高质量轨迹的微调（Hint‑RFT）。在推理阶段，模型像普通 LRM 那样输出思维链，但每当出现预设的提示词，系统会截取后面的代码或表达式交给外部解释器执行；执行结果被插回思维链，模型继续往下写。微调阶段则把这些完整的“思维链+工具结果”当作训练样本，进一步强化模型的工具感知能力。

**关键模块拆解**  

1. **Hint‑infer 触发器**  
   - 在模型的输出流中预置若干触发词（如 “使用 Python”, “运行代码”, “计算一下”）。  
   - 当模型生成这些词时，系统立即解析后面的代码块或数学表达式。  
   - 类比：就像在写笔记时写下“TODO: 用计算器算”，随后打开计算器完成计算。

2. **工具执行引擎**  
   - 当前实现主要是 Python 解释器，能够执行数值运算、绘图、甚至调用常用库（NumPy、SymPy）。  
   - 执行后返回的文本（数值、错误信息）被包装成特殊标记，插回模型的输出流。  
   - 设计上保持了“安全沙箱”，防止恶意代码影响主进程。

3. **自检与自调试回路**  
   - 模型在收到工具输出后，会生成一段自检文字，如 “检查结果是否与前一步推理一致”。  
   - 若检测到不一致（比如除零错误、符号错误），模型会生成新的代码或修改提示，再次进入工具执行。  
   - 这一步类似人类在做实验后重复验证的过程。

4. **Hint‑RFT 微调管线**  
   - 收集大量 Hint‑infer 产生的完整轨迹。  
   - 用一个评分模型（基于答案正确率和轨迹流畅度）挑选前 10% 的高质量样本。  
   - 对原始 LRM（这里是 QwQ‑32B）进行有监督微调，目标是让模型在没有显式提示的情况下也能自行产生相同的工具调用模式。  
   - 关键在于“拒绝采样”（Rejection Sampling）：低质量的轨迹直接被丢弃，防止噪声污染微调数据。

**最巧妙的设计**  
Hint‑infer 把“工具使用”从显式指令变成了隐式思维链的一部分，几乎不需要额外的标注成本；而 Hint‑RFT 则把这种隐式能力显式化，形成了自我强化的闭环。两者结合，使得模型在推理时既能像人一样“写草稿”，又能像程序员一样“跑代码”，而不需要人为手动标注每一步的工具调用。

### 实验与效果
- **测试任务**：  
  - GPQA（PhD 级科学问答）  
  - AMC23、AIME24、AIME25（美国数学竞赛）  
  - LiveCodeBench（代码生成与执行）  

- **主要结果**（相较于未使用工具的 QwQ‑32B）：  
  - GPQA 正确率从原始的约 45% 提升到 **63.6%**。  
  - AMC23 达到 **95.0%**，AIME24 为 **66.7%**，AIME25 为 **47.1%**。  
  - LiveCodeBench 准确率提升至 **47.3%**。  

- **对标**：这些数字与闭源的 o1‑Preview、以及开源的 R1‑Distill‑Qwen‑32B 相当，说明 START 在不依赖大规模专有数据的情况下也能逼近最前沿水平。

- **消融实验**（原文提供）：  
  - 去掉 Hint‑infer，模型几乎不调用工具，性能回落到原始 LRM 水平。  
  - 只做 Hint‑RFT 而不进行实际工具执行，提升有限，说明真实工具反馈是关键。  
  - 自检/自调试环节的加入，使错误率下降约 12%，验证了循环校验的价值。

- **局限性**：  
  - 目前仅支持 Python，面对需要专用软件（如 CAD、化学模拟）的任务仍然受限。  
  - 提示词的设计仍是手工规则，跨语言或跨领域的自动化提示生成尚未探索。  
  - 大规模微调成本仍高，模型在资源受限的环境下的部署需要进一步压缩。

### 影响与延伸思考
START 把“语言模型 + 工具”从“外部插件”提升到“内部思维链的自然组成部分”，在学术界和工业界引发了两股潮流：一是围绕 **提示驱动的工具激活**（如 AutoCoT、Toolformer 的后续工作）展开的研究；二是 **自监督微调** 通过生成式轨迹提升模型工具感知的探索（如 ReAct、Self‑Ask 的进化版）。后续工作可能会在以下方向继续深化：  
- 自动化生成提示词的元学习方法（推测）。  
- 多模态工具集成（图像处理、物理仿真）以覆盖更广的科学任务。  
- 将 Hint‑RFT 与 RLHF（基于人类反馈的强化学习）结合，进一步提升安全性和可解释性。  

如果想深入了解，可以关注 **Toolformer**、**ReAct**、以及最近的 **Self‑Debugging LLM** 系列论文，它们在不同维度上都受到了 START 思路的启发。

### 一句话记住它
让大模型在思考时主动“叫外卖”，用简短提示诱导工具调用，再把成功的点单记录下来，模型自己学会了“写代码+检查答案”。
