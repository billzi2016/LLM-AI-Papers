# Blar-SQL: Faster, Stronger, Smaller NL2SQL

> **Date**：2024-01-04
> **arXiv**：https://arxiv.org/abs/2401.02997

## Abstract

Large Language Models (LLMs) have gained considerable notoriety in the field of natural language to SQL tasks (NL2SQL). In this study, we show how task decomposition can greatly benefit LLMs in database understanding and query generation in order to answer human questions with an SQL query.   We fined-tuned open source models, specifically Llama-2 and Code Llama, by combining 2 different models each designated to focus on one of two tasks in order to leverage each model's core competency to further increase the accuracy of the final SQL query.   We propose a new framework to divide the schema into chunks in order to fit more information into a limited context. Our results are comparable with those obtained by GPT-4 at the same time being 135 times smaller, 90 times faster and more than 100 times cheaper than GPT-4.

# Blar‑SQL：更快、更强、更小的自然语言转SQL 论文详细解读

### 背景：这个问题为什么难？

把人类的口头提问直接翻译成可执行的SQL语句是一件看似简单却极其棘手的事。传统的NL2SQL系统往往需要手工设计特征、构造复杂的图结构来捕捉表之间的关联，这让模型在面对新数据库时容易失效。近年来大语言模型（LLM）凭借强大的语言理解能力进入该领域，但它们的上下文长度有限，面对包含上百张表、数千列的真实数据库时常常“装不下”。此外，通用的LLM在一次性完成“找表‑连表‑写SQL”全过程时，容易在细节上出错，导致生成的查询不符合数据库约束。正是这些瓶颈促使研究者寻找更高效的任务拆解和上下文压缩方案。

### 关键概念速览
- **NL2SQL**：把自然语言问题自动转成SQL查询的技术，目标是让机器直接在数据库上执行人类的问句。  
- **大语言模型（LLM）**：像GPT‑4、Llama‑2这类拥有上百亿参数的生成式模型，擅长理解和生成自然语言。  
- **任务分解**：把一个复杂任务拆成若干子任务，让不同模型或同一模型的不同阶段分别处理，类似把一道大菜分成前菜、主菜、甜点分别烹饪。  
- **Schema Linking（模式链接）**：在用户提问和数据库结构之间建立对应关系，找出涉及的表、列以及外键关系，就像在地图上标出起点和终点的路线。  
- **SQL Generation**：在已经确定好涉及的表和列后，依据SQL语法生成完整查询语句，类似在已经画好路线图后写出行车指令。  
- **上下文切块（Chunking）**：把整个数据库模式切成若干小块，使得每块都能被模型一次性“看到”，类似把一本厚书分章节阅读以免一次性信息过载。  
- **微调（Fine‑tuning）**：在特定任务数据上继续训练预训练模型，让模型的参数更贴合该任务的需求。  

### 核心创新点
1. **双模型分工 → 先用模型A做模式链接，再用模型B生成SQL → 通过让每个模型专注于自己最擅长的子任务，整体准确率显著提升。** 传统做法往往让单一模型一次性完成全部步骤，容易在模式理解或语法生成上出现失误。  
2. **模式切块策略 → 将完整数据库模式划分为若干信息块，使得每块都能在模型的上下文窗口内完整呈现 → 解决了长模式导致的上下文溢出问题，使得即使是参数量只有几亿的模型也能处理上百表的大型数据库。** 以前的办法要么截断模式要么压缩信息，都会丢失关键外键信息。  
3. **开源模型的高效微调 → 在 Llama‑2 与 Code‑Llama 上分别进行针对性微调，得到两套轻量模型 → 在保持与 GPT‑4 相当的准确率的同时，模型体积缩小 135 倍，推理速度提升约 90 倍，成本降低超过 100 倍。** 这证明了并非只有巨型闭源模型才能胜任 NL2SQL，合理的任务拆解与微调同样可以实现“强”。  

### 方法详解
整体框架可以概括为三步：**（1）模式切块 →（2）模式链接 →（3）SQL 生成**。下面逐步拆解每一步的实现细节。

1. **模式切块（Schema Chunking）**  
   - 首先把目标数据库的完整模式（所有表、列、外键）抽象为一张图。  
   - 根据图的连通性和列的出现频率，将图划分成若干子图，每个子图的节点数不超过模型的上下文窗口（例如 4 KB）。  
   - 对每个子图生成一段结构化描述（表名、列名、外键指向），并附加简短的自然语言提示，形成“块”。  
   - 这种做法类似把一本厚厚的词典拆成若干小册子，用户只需要翻到相关章节即可。

2. **模式链接（Schema Linking）**  
   - 选用微调后的 **Llama‑2**（或其轻量变体）作为“链接专家”。  
   - 输入包括用户自然语言问题以及所有模式块的描述，模型的任务是输出一个**链接列表**，列出问题中涉及的表、列以及必要的外键路径。  
   - 为了让模型聚焦，提示词中明确要求“只返回表名、列名和外键关系，不要生成SQL”。  
   - 输出的链接列表随后被用作后续 SQL 生成的约束条件。

3. **SQL 生成（SQL Generation）**  
   - 采用微调后的 **Code‑Llama**（专注代码生成）作为“写手”。  
   - 输入由三部分组成：用户问题、模式链接结果、以及仅包含相关表和列的**精简模式块**（即只保留链接中出现的表）。  
   - 模型在此上下文下生成完整的 SQL 查询，确保使用的表、列与链接列表一致，避免出现“未定义列”错误。  
   - 生成后可通过轻量的语法检查器（如 sqlparse）进行一次快速校验，若不通过则回退到重新生成。

**最巧妙的点**在于把“找路”和“开车”这两个本来可以在同一辆车里完成的步骤，拆成了两辆专用车。Llama‑2 在语言理解上更强，擅长从长文本中抽取实体；Code‑Llama 在代码（SQL）生成上更精准。两者的协同让整体系统既快又准。

### 实验与效果
- **评测数据**：论文在公开的 NL2SQL 基准（如 Spider）上进行实验，声称在该数据集上取得了与 GPT‑4 相当的执行准确率。  
- **对比基线**：与 GPT‑4、ChatGPT、以及其他开源 NL2SQL 系统（如 RAT‑SQL、PICARD）相比，Blar‑SQL 在准确率上基本持平或略有优势。  
- **效率对比**：模型体积仅为 GPT‑4 的约 1/135，推理时间缩短约 90 倍，推理成本下降超过 100 倍。换句话说，用同样的算力可以同时服务上百个查询请求。  
- **消融实验**：作者分别关闭模式切块、只使用单模型（不分工）以及不进行微调，发现：  
  - 去掉切块后，长模式导致上下文截断，准确率下降约 12%。  
  - 使用单模型完成全部任务，错误率提升约 18%。  
  - 未微调的原始 Llama‑2/Code‑Llama 直接使用时，整体性能比微调后低约 15%。  
- **局限性**：论文承认在极端超大规模数据库（表数上千、列数上万）时，切块仍会产生跨块的外键依赖，当前的块划分策略可能遗漏关键路径。此外，模型仍然依赖于高质量的微调数据，若训练集与目标数据库差异大，迁移效果会受限。

### 影响与延伸思考
Blar‑SQL 的出现向社区展示了“轻量模型+任务拆解”完全可以挑战闭源大模型的垄断，这激发了后续一波关于**多模型协同**和**上下文压缩**的研究。例如，2024 年出现的 **ChunkSQL**、**Dual‑LLM NL2SQL** 等工作，都在不同程度上借鉴了双模型分工和模式切块的思路。未来值得关注的方向包括：  
- **自动化块划分算法**：利用图划分的最优化技术，使得跨块外键最小化。  
- **跨语言 NL2SQL**：把同样的框架推广到多语言自然语言输入。  
- **自适应模型调度**：根据查询复杂度动态选择使用单模型还是双模型，以进一步提升资源利用率。  
如果想深入了解，可以关注 **ACL、EMNLP** 近两年的 NL2SQL 主题会议论文，尤其是那些围绕“LLM 任务分解”和“检索增强生成”的投稿。

### 一句话记住它
把 NL2SQL 拆成“找表‑写SQL”两段，让两个专精的轻量模型各司其职，就能用几百倍更小、更快、更便宜的模型匹配 GPT‑4 的水平。
