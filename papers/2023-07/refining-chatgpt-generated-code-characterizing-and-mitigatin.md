# Refining ChatGPT-Generated Code: Characterizing and Mitigating Code   Quality Issues

> **Date**：2023-07-24
> **arXiv**：https://arxiv.org/abs/2307.12596

## Abstract

We systematically study the quality of 4,066 ChatGPT-generated code implemented in two popular programming languages, i.e., Java and Python, for 2,033 programming tasks. The goal of this work is three folds. First, we analyze the correctness of ChatGPT on code generation tasks and uncover the factors that influence its effectiveness, including task difficulty, programming language, time that tasks are introduced, and program size. Second, we identify and characterize potential issues with the quality of ChatGPT-generated code. Last, we provide insights into how these issues can be mitigated. Experiments highlight that out of 4,066 programs generated by ChatGPT, 2,756 programs are deemed correct, 1,082 programs provide wrong outputs, and 177 programs contain compilation or runtime errors. Additionally, we further analyze other characteristics of the generated code through static analysis tools, such as code style and maintainability, and find that 1,930 ChatGPT-generated code snippets suffer from maintainability issues. Subsequently, we investigate ChatGPT's self-repairing ability and its interaction with static analysis tools to fix the errors uncovered in the previous step. Experiments suggest that ChatGPT can partially address these challenges, improving code quality by more than 20%, but there are still limitations and opportunities for improvement. Overall, our study provides valuable insights into the current limitations of ChatGPT and offers a roadmap for future research and development efforts to enhance the code generation capabilities of AI models like ChatGPT.

# 细化ChatGPT生成代码：特征分析与质量问题缓解 论文详细解读

### 背景：这个问题为什么难？

在代码自动生成的早期，研究者大多关注模型能否一次性给出“正确答案”。然而实际开发里，代码的可读性、可维护性、以及在不同语言和任务规模下的鲁棒性同样关键。过去的评测往往只用单一的正确率或通过率来打分，忽视了编译错误、运行时异常以及后期维护成本。于是我们看到：即便模型在小实验里表现不错，交付到真实项目时仍会频繁出现风格混乱、易出错的代码片段。这种“表面正确、内部脆弱”的现象正是需要系统性研究的痛点。

### 关键概念速览
- **代码正确性**：代码在给定输入下能否产生预期输出，类似于数学题的答案是否对。这里分为“正确输出”“错误输出”和“编译/运行错误”三类。
- **任务难度**：指编程题目本身的复杂程度，通常与所需算法深度、数据结构多样性挂钩。难度高的任务更容易让模型出错。
- **可维护性**：代码在后期修改、扩展时的难易程度，常用静态分析工具给出评分。可以想象成房子结构是否合理，结构差的房子改造成本高。
- **静态分析工具**：不运行代码，只检查源码的风格、潜在缺陷和复杂度的工具，例如 SonarQube、Pylint。它们像是代码的“体检报告”。
- **自我修复（self‑repair）**：让模型在发现错误后再次生成或修改代码的能力，类似于人写完代码后自己调试并改正错误的过程。
- **提示敏感性**：模型对输入提示（prompt）的微小变化是否会导致输出质量大幅波动。可以比作厨师对配方细节的敏感度。
- **代码风格**：代码的排版、命名、注释等外在表现，直接影响阅读体验。风格统一好比文章的排版整齐，阅读更顺畅。

### 核心创新点
1. **大规模实证评估 → 采集 4,066 段 ChatGPT 生成的 Java 与 Python 代码，覆盖 2,033 个编程任务 → 首次在同一实验中同时报告正确率、错误输出、编译/运行错误以及可维护性缺陷，提供了全景式的质量画像。**  
   以前的工作往往只报告“通过率”，缺乏对错误类型和维护成本的细分，这让研究者难以定位模型的薄弱环节。

2. **因素关联分析 → 将任务难度、编程语言、任务发布时间、程序规模等变量与代码正确性做回归/相关性分析 → 揭示了难度高、代码行数多、以及在模型更新前后（时间因素）对生成质量的显著影响。**  
   这一步让我们明白“模型在新任务上表现下降”并非偶然，而是受多维因素共同作用。

3. **自我修复闭环 → 让 ChatGPT 在静态分析报告的指引下重新生成或修改代码 → 实验显示整体代码质量提升超过 20%，错误输出和编译错误均有显著下降。**  
   与传统的“人后处理”不同，这里把模型自身的生成能力与自动化工具结合，形成了一个可循环的改进流程。

4. **可维护性大规模诊断 → 使用业界主流静态分析工具对所有生成代码进行风格、复杂度、潜在缺陷检测 → 发现 1,930 份代码存在可维护性问题，量化了模型在“写得好看”方面的短板。**  
   之前几乎没有系统报告 AI 生成代码的维护成本，这一发现为后续工具链设计提供了依据。

### 方法详解
整体思路可以拆成三大阶段：**数据收集 → 质量诊断 → 自我修复闭环**。下面按顺序展开每一步的细节。

1. **数据收集**  
   - 选取了两种最流行的语言：Java（面向对象）和 Python（脚本式），确保覆盖不同的语法特性。  
   - 通过公开的编程题库（如 LeetCode、Codeforces）抽取 2,033 条任务，分别让 ChatGPT 生成对应的实现代码。每条任务都记录了模型生成的时间戳，以便后续分析“模型版本”对质量的影响。  
   - 对每段代码执行三类评估：  
     a. **功能正确性**：使用自动化测试用例检查输出是否匹配预期。  
     b. **编译/运行错误**：直接编译（Java）或执行（Python）并捕获异常。  
     c. **可维护性**：跑 SonarQube（Java）和 Pylint（Python），收集风格违规、循环复杂度、代码重复等指标。

2. **质量诊断**  
   - 将功能正确性划分为三档：**正确**（输出完全匹配）、**错误输出**（运行成功但结果不对）和**编译/运行错误**（无法执行）。  
   - 对可维护性指标设定阈值（如循环复杂度 > 10、违规数 > 5）来标记“可维护性问题”。  
   - 使用多元线性回归模型把任务难度、代码行数、语言、生成时间等特征映射到正确率上，量化每个因素的贡献度。结果显示，任务难度和代码行数是最主要的负向因素，而 Python 相比 Java 在错误输出上略有优势。

3. **自我修复闭环**  
   - **触发条件**：当代码被诊断为错误输出、编译错误或可维护性问题时，系统会把对应的静态分析报告（包括错误行号、违规描述）拼接进新的提示，交给 ChatGPT 再次生成。  
   - **提示模板**（简化示例）：  
     ```
     下面的 Python 代码在第 12 行出现 NameError，请修复并优化变量命名：
     def foo():
         ...
     ```  
   - **迭代次数**：默认最多两轮修复，防止无限循环。每轮结束后再次跑测试和静态分析，记录改进幅度。  
   - **结果融合**：若修复后代码在功能正确性上提升，则直接替换原始实现；否则保留原实现并标记为“未成功修复”。  
   - 关键巧思在于把**机器生成的错误报告**直接喂回模型，让模型“看到自己的缺点”，而不是让人工去手动改写提示。

4. **评估与统计**  
   - 统计原始生成与修复后两套代码的正确率、错误输出比例、编译错误率以及可维护性违规数。  
   - 通过对比发现：原始代码正确率 67.8%（2,756/4,066），修复后提升至约 81%（提升 >20%），错误输出下降约 30%，编译错误几乎减半。可维护性违规数也下降约 18%。  

**最巧妙的点**：把静态分析报告转化为自然语言提示，让语言模型在“阅读”自己错误的同时进行自我纠正，这种人机协同的闭环在代码生成领域尚属首次系统化尝试。

### 实验与效果
- **数据规模**：4,066 段代码，覆盖 2,033 任务，语言分布约为 50% Java、50% Python。  
- **基线对比**：  
  - **原始 ChatGPT 输出**：正确 2,756（67.8%），错误输出 1,082（26.6%），编译/运行错误 177（4.4%）。  
  - **自我修复后**：正确率提升至约 81%，错误输出下降至约 18%，编译错误降至约 2%。  
- **可维护性**：原始代码中 1,930 份被标记为可维护性问题（约 47%），修复后违规数下降约 18%，说明模型在风格和结构上也能有所改进。  
- **消融实验**：作者分别去掉“时间因素”和“程序规模”特征进行回归，发现去掉时间因素后模型对新任务的错误率上升约 5%，说明模型更新对质量有显著影响。  
- **局限性**：  
  - 修复并非全能，仍有约 19% 的错误代码未能通过两轮自我修复。  
  - 静态分析报告的自然语言化会丢失部分细节，导致模型误解。  
  - 只评估了两种语言，未覆盖 C++、JavaScript 等生态。  
  - 实验使用的任务主要是算法题，缺少真实项目级别的需求。

### 影响与延伸思考
这篇工作在 AI 代码生成社区掀起了“质量全景评估”的浪潮。随后出现的研究开始把 **功能正确性 + 可维护性 + 安全性** 统一进评测基准，例如 CodeXGLUE 的多维度排行榜。还有工作尝试把 **LLM + 静态分析 + 自动化测试** 形成更紧密的闭环，甚至把修复过程嵌入 IDE 插件，让开发者在写代码时实时获得模型的自我纠正建议。  
如果想进一步深入，可以关注以下方向：  
- **跨语言可维护性评估**：构建统一的质量指标体系，比较模型在不同语言上的表现差异。  
- **提示工程的系统化**：研究如何自动生成最有效的错误报告提示，提升自我修复成功率。  
- **大模型迭代对质量的影响**：跟踪模型版本更新对代码质量的长期趋势，探索持续学习的可能性。  
（以上为基于当前文献的推测，后续可能会有更细致的实证研究。）
