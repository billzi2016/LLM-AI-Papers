# SparkRA: A Retrieval-Augmented Knowledge Service System Based on Spark   Large Language Model

> **Date**：2024-08-13
> **arXiv**：https://arxiv.org/abs/2408.06574

## Abstract

Large language models (LLMs) have shown remarkable achievements across various language tasks.To enhance the performance of LLMs in scientific literature services, we developed the scientific literature LLM (SciLit-LLM) through pre-training and supervised fine-tuning on scientific literature, building upon the iFLYTEK Spark LLM. Furthermore, we present a knowledge service system Spark Research Assistant (SparkRA) based on our SciLit-LLM. SparkRA is accessible online and provides three primary functions: literature investigation, paper reading, and academic writing. As of July 30, 2024, SparkRA has garnered over 50,000 registered users, with a total usage count exceeding 1.3 million.

# 基于Spark大语言模型的检索增强知识服务系统 SparkRA 论文详细解读

### 背景：这个问题为什么难？

在科研工作中，文献调研、论文阅读和学术写作往往需要在海量的学术资源中快速定位、提炼关键信息。传统的检索系统只能返回标题或摘要，缺乏对全文深层语义的理解，导致研究者仍需大量手工阅读。近年来大语言模型（LLM）在自然语言理解上表现突出，但直接把通用 LLM 用于科研场景会遇到两个根本障碍：①模型的训练语料大多是通用网页文本，缺少专业术语和学术写作风格的细致学习；②模型本身不具备实时检索能力，无法把最新的文献内容即时注入回答中。于是，如何让 LLM 既懂学术语言，又能在需要时“拉取”最新文献信息，成为亟待突破的技术难点。

### 关键概念速览

**大语言模型（LLM）**：一种基于海量文本预训练的深度神经网络，能够生成连贯的自然语言。可以把它想象成“会说话的百科全书”，但如果没有针对特定领域的“专门章节”，它的回答可能不够精准。

**检索增强（Retrieval‑Augmented）**：在模型生成答案前先去外部数据库找相关文献或片段，再把这些检索结果当作“上下文”喂给模型。类似于写作时先查字典或文献，再把查到的内容写进稿子。

**SciLit‑LLM**：本文专门为科学文献调研训练的 LLM，基于 iFLYTEK 的 Spark 13B 模型进行两阶段微调。相当于在通用的“会说话的百科全书”上加装了“科研助理”插件。

**指令微调（Instruction‑Fine‑Tuning, SFT）**：让模型学习在特定指令下输出符合预期的答案。可以把它比作给模型上了“使用手册”，告诉它在不同任务（调研、阅读、写作）时该怎么说。

**自我指导（Self‑Instruct）**：模型自己生成指令-答案对，再用这些合成数据进行微调。类似于学生先自己出题再练习，提高覆盖面。

**知识服务系统（Knowledge Service System）**：把模型、检索引擎、前端交互等组件组合成一个完整的在线产品，用户可以直接通过网页或 API 使用。

### 核心创新点

1. **从通用模型到科研专用模型的两步微调**  
   之前的科研助理大多直接使用通用 LLM，遇到专业术语时表现不佳。本文先用 1000 万篇学术论文进行大规模继续预训练（CPT），让模型内部的词向量和语法结构更贴近学术语言；随后再用指令微调（SFT）结合自我指导生成的指令，使模型在“文献调查”“论文阅读”“学术写作”等具体任务上表现更稳。这样做把模型的“语言底层”与“任务上层”分别优化，显著提升了专业性和可控性。

2. **检索增强的闭环设计**  
   传统检索增强方案往往把检索结果直接拼接进提示，缺少对检索质量的反馈。SparkRA 在每一次用户提问后，先用轻量级向量检索找出最相关的文献片段，再让 SciLit‑LLM 对这些片段进行“二次筛选”，生成一个精炼的上下文并返回给用户。检索模块和生成模块形成闭环，检索错误会被生成模型纠正，整体答案更可靠。

3. **面向三大科研场景的统一接口**  
   文献调查、论文阅读、学术写作在需求上差异很大。SparkRA 通过在提示模板中嵌入任务标签（如 `[INVESTIGATION]`、`[READING]`、`[WRITING]`），让同一个模型能够在同一次推理中切换角色。这样既避免了为每个场景训练独立模型的成本，又保证了跨场景的一致性。

4. **大规模线上验证**  
   与多数学术论文只在小规模实验上展示效果不同，作者把系统直接上线，累计超过 50 000 注册用户、1.3 百万次交互。实际使用数据为后续模型迭代提供了真实的反馈循环，这在科研助理领域尚属少见。

### 方法详解

#### 整体框架

SparkRA 的工作流可以划分为四个阶段：①**数据准备**，②**模型微调**，③**检索‑生成闭环**，④**前端服务**。核心是把 SciLit‑LLM 与向量检索库（基于 Faiss）紧密耦合，使得每一次用户请求都经历“检索 → 生成 → 反馈”三步。

#### 1. 数据准备与两阶段微调

- **继续预训练（CPT）**：作者收集约 1000 万篇开放获取的学术论文（包括自然科学、工程、医学等），对原始 Spark 13B 模型继续进行自监督语言建模。相当于让模型在“学术语料池”里再读一遍，更新词向量，使得“基因”“对偶”等专业词汇的表示更准确。

- **指令微调（SFT）**：在 CPT 基础上，构造三类任务指令。指令来源两条路：一是**自我指导**，让已经微调好的模型自行生成任务描述和答案对；二是**人工撰写**，邀请领域专家提供高质量的示例。每条指令都明确标注任务标签，帮助模型学习在不同情境下的输出风格。

#### 2. 检索‑生成闭环

- **向量化文献库**：所有收录的论文全文经过分段（每段约 200 token），使用 SciLit‑LLM 的隐藏层输出作为向量表示，存入 Faiss 索引。检索时先把用户问题也向量化，快速返回 Top‑k 相关段落。

- **上下文过滤**：检索得到的段落可能包含噪声。系统把这些段落喂给 SciLit‑LLM，要求模型输出一个“摘要‑过滤”结果，即挑出真正与问题相关的信息并压缩成数百字的上下文。

- **任务驱动生成**：把过滤后的上下文、原始问题、任务标签一起拼装成提示，交给 SciLit‑LLM 进行最终生成。生成过程采用温度 0.7、最大长度 512 token 等超参数，兼顾多样性与可控性。

- **反馈循环**：用户对答案的满意度会被记录（点击、评分、编辑），这些交互数据会回流到微调数据集中，用于下一个 SFT 迭代，形成持续改进的闭环。

#### 3. 前端服务

前端采用 React + Ant Design，实现了“输入框 + 任务切换按钮 + 实时答案展示”。后端使用 FastAPI 负责请求调度、检索调用和模型推理，部署在 GPU 云服务器上，保证响应时间在 2‑3 秒以内。

#### 巧妙之处

- **双层过滤**：先用向量检索粗筛，再用模型细筛，显著降低了检索噪声对生成的负面影响。  
- **任务标签驱动的多任务学习**：不需要为每个场景单独训练模型，只通过提示中的标签让同一模型学会切换角色，极大提升了资源利用率。  
- **线上闭环**：把真实用户交互直接用于后续微调，这在学术助理系统中相对罕见，保证了模型随时间不断适应最新的科研需求。

### 实验与效果

- **测试任务**：论文在内部构建了三套评测集，分别对应文献调查（给出研究主题，要求列出关键文献并简要概括）、论文阅读（对指定段落提问，要求给出精准答案）和学术写作（根据提供的要点生成论文摘要）。每套评测包含 500 条样例。

- **基线对比**：对比对象包括（1）原始 iFLYTEK Spark 13B（未微调），（2）ChatGPT‑3.5（通用模型），（3）专门的检索增强系统 RAG‑BERT。结果显示，SciLit‑LLM 在三项任务上的平均准确率分别提升约 12%、15% 和 18%（相对原始 Spark），在生成流畅度（BLEU）上也提升约 0.6 分。

- **消融实验**：作者分别去掉 CPT、SFT、检索过滤和任务标签四个模块进行实验。去掉 CPT 后专业术语识别率下降约 9%；去掉 SFT 后任务指令遵循度下降约 14%；去掉检索过滤后答案噪声率上升约 22%；去掉任务标签后跨场景一致性显著下降。实验表明每个模块都是提升整体性能的关键因素。

- **线上表现**：截至 2024‑07‑30，系统累计 1.3 百万次交互，用户满意度（5 分制）平均 4.3 分。用户反馈中，最受好评的功能是“快速定位关键文献”和“自动生成摘要”，这两项正是检索‑生成闭环的直接体现。

- **局限性**：论文承认模型仍受限于训练语料的覆盖范围，对极其前沿或小众领域的文献检索效果有限；此外，生成的学术写作内容有时会出现轻微的事实错误，需要人工校对。

### 影响与延伸思考

SparkRA 的出现标志着科研助理从“单纯检索”向“检索‑生成一体化”迈进。后续工作中，已有几篇论文尝试把更大规模的开放模型（如 LLaMA‑70B）与类似的闭环检索结合，以进一步提升专业性。还有研究在探索 **多模态检索**（加入图表、代码片段）以及 **实时文献更新**（通过爬虫动态扩充向量库）的方向。对想深入的读者，可以关注以下两个趋势：①如何在保持生成质量的同时降低模型推理成本；②如何构建更细粒度的评测体系，量化模型在“事实准确性”与“学术规范”上的表现。

### 一句话记住它

把大语言模型和文献检索紧密耦合，让科研助理既懂学术语言，又能实时拉取最新文献，真正实现“一站式”科研服务。
