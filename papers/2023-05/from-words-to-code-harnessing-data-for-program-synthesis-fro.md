# From Words to Code: Harnessing Data for Program Synthesis from Natural   Language

> **Date**：2023-05-02
> **arXiv**：https://arxiv.org/abs/2305.01598

## Abstract

Creating programs to correctly manipulate data is a difficult task, as the underlying programming languages and APIs can be challenging to learn for many users who are not skilled programmers. Large language models (LLMs) demonstrate remarkable potential for generating code from natural language, but in the data manipulation domain, apart from the natural language (NL) description of the intended task, we also have the dataset on which the task is to be performed, or the "data context". Existing approaches have utilized data context in a limited way by simply adding relevant information from the input data into the prompts sent to the LLM.   In this work, we utilize the available input data to execute the candidate programs generated by the LLMs and gather their outputs. We introduce semantic reranking, a technique to rerank the programs generated by LLMs based on three signals coming the program outputs: (a) semantic filtering and well-formedness based score tuning: do programs even generate well-formed outputs, (b) semantic interleaving: how do the outputs from different candidates compare to each other, and (c) output-based score tuning: how do the outputs compare to outputs predicted for the same task. We provide theoretical justification for semantic interleaving. We also introduce temperature mixing, where we combine samples generated by LLMs using both high and low temperatures. We extensively evaluate our approach in three domains, namely databases (SQL), data science (Pandas) and business intelligence (Excel's Power Query M) on a variety of new and existing benchmarks. We observe substantial gains across domains, with improvements of up to 45% in top-1 accuracy and 34% in top-3 accuracy.

# 从文字到代码：利用数据进行自然语言程序合成 论文详细解读

### 背景：这个问题为什么难？
在数据处理场景下，用户往往只会用自然语言描述想要的操作，却不熟悉 SQL、Pandas 或 Power Query 等专业语言。传统的代码生成模型只能把文字直接翻译成代码，忽略了手头实际数据的“上下文”。于是模型经常产生语法正确但在真实数据上跑不通的程序。之前的做法只是在提示（prompt）里把几行样本数据拼进去，效果提升有限，因为模型仍然缺乏对候选程序执行结果的直接反馈。

### 关键概念速览
**自然语言到代码（NL→Code）**：把用户的文字需求转化为可执行的程序代码，类似把口头指令翻译成机器指令。  
**数据上下文（Data Context）**：任务所针对的具体数据表或文件，模型可以直接读取的原始输入。把它想象成厨师手里的食材，只有知道食材才能做出合适的菜。  
**候选程序（Candidate Program）**：LLM 在一次采样中生成的代码片段，可能有多个备选。  
**语义重排（Semantic Reranking）**：根据程序运行后得到的输出，对所有候选程序重新排序的过程，就像先做菜再尝味道，味道好坏决定最终上桌的菜。  
**温度混合（Temperature Mixing）**：在生成代码时使用不同的采样温度（高温产生多样性、低温产生确定性），然后把所有结果混在一起评估。  
**语义过滤（Semantic Filtering）**：检查程序输出是否符合基本格式或类型要求，过滤掉明显错误的候选。  
**语义交叉（Semantic Interleaving）**：比较不同候选程序的输出之间的相似度或差异，用来判断哪类答案更可信。  

### 核心创新点
1. **执行驱动的反馈回路**：以前的系统只把数据当作提示文字，缺少实际运行的环节。这里先让 LLM 生成若干代码，再在真实数据上执行，收集输出。这样模型的“猜测”被真实结果检验，显著提升了正确率。  
2. **三维语义重排**：在得到输出后，论文提出了三种信号来给每个候选程序打分：① 检查输出是否结构完整（语义过滤），② 比较不同候选之间的输出一致性（语义交叉），③ 与任务的预期输出模型进行对比（输出基准调优）。这套评分体系把“代码好不好”量化为可比较的数值。  
3. **温度混合采样**：单一温度往往要么太保守要么太随机。作者同时采样高温和低温的代码，然后统一交给语义重排，这相当于让模型既提供创新思路，又保留稳妥答案，最终提升了 top‑1 与 top‑3 的命中率。  
4. **跨语言统一框架**：方法在 SQL、Pandas、Power Query 三种完全不同的编程环境下都能直接套用，证明了“执行+重排”不是针对某一语言的特例，而是一种通用的程序合成思路。

### 方法详解
整体思路可以拆成四步：  
1. **提示构造**：把用户的自然语言需求和少量数据样本拼成 Prompt，送给大语言模型（LLM）。  
2. **多样化采样**：在两个温度设置（高、低）下各采样若干次，得到一批候选程序。  
3. **执行与收集**：把每段代码在对应的数据环境中运行，捕获输出或错误信息。  
4. **语义重排**：基于三类信号为每个候选打分，重新排序后返回最高分的代码。

**关键模块细化**  
- **提示层**：除了原始需求，还会加入数据的列名、示例行等元信息，让模型对数据结构有基本认知。  
- **温度混合器**：高温采样倾向于产生结构多变的代码（比如不同的 JOIN 方式），低温采样则更可能复刻训练中常见的模式。两者的输出集合被统一送入后续评估。  
- **执行引擎**：为每种语言准备沙箱环境（SQL 用 SQLite，Pandas 用 Pandas DataFrame，Power Query 用 M 引擎），确保代码安全且能快速返回结果。  
- **语义过滤器**：检查输出是否符合预期的类型（比如返回表格而不是单个数字），并剔除运行时报错的候选。  
- **语义交叉器**：计算不同候选输出之间的相似度，例如使用集合相似度或行列匹配度。如果多数候选给出相近结果，则该结果的可信度提升。  
- **输出基准调优**：作者训练了一个轻量的预测模型，估计在相同任务下理想的输出形态，用来对实际输出进行打分。  

**最巧妙的点**  
- 直接把“执行结果”当作评分依据，而不是仅靠语言模型的内部置信度，这让系统能够纠正模型的语法或逻辑错误。  
- 语义交叉的理论证明表明，当多数候选输出相似时，错误概率指数级下降，这为多样化采样提供了数学支撑。  

### 实验与效果
- **测试领域**：SQL（数据库查询）、Pandas（数据科学脚本）和 Power Query M（商业智能）。使用了公开的 Spider、WikiSQL、以及作者自行构建的 Pandas 与 Power Query 基准。  
- **对比基线**：传统的 NL→Code 系统（仅提示+LLM）、加入数据提示的增强版、以及最近的基于检索的代码生成模型。  
- **提升幅度**：在 SQL 上 top‑1 准确率提升约 30%，top‑3 提升约 25%；在 Pandas 上 top‑1 提升最高 45%，top‑3 提升约 34%；Power Query 也有类似的两位数提升。  
- **消融实验**：去掉温度混合后 top‑1 下降约 8%；仅保留语义过滤不使用交叉和基准调优，准确率下降约 12%；完全不执行直接返回 LLM 输出，性能回到原始基线。说明每个模块都有实质贡献。  
- **局限性**：执行需要安全沙箱，成本随数据规模线性增长；对极其大型数据集或实时流式任务仍不适用；论文未在代码生成的长程序（上千行）上做评估，可能受限于采样数量。  

### 影响与延伸思考
这篇工作把“执行-反馈”引入自然语言代码生成，开启了“程序合成即验证” 的新潮流。随后的研究开始探索更高效的执行代理、利用差分执行结果进行梯度微调，甚至把执行过程本身嵌入到 LLM 的内部推理图中（如“自检式 LLM”）。如果想进一步了解，可以关注以下方向：① 更轻量的沙箱技术（如 WebAssembly），② 基于执行轨迹的强化学习微调，③ 多模态数据上下文（图像+表格）对代码生成的影响。  

### 一句话记住它
把候选代码跑一遍，用输出的好坏来给它们打分，代码生成的准确率立马飙升。
