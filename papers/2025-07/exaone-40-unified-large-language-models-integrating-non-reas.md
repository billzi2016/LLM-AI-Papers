# EXAONE 4.0: Unified Large Language Models Integrating Non-reasoning and Reasoning Modes

> **Date**：2025-07-15
> **arXiv**：https://arxiv.org/abs/2507.11407

## Abstract

This technical report introduces EXAONE 4.0, which integrates a Non-reasoning mode and a Reasoning mode to achieve both the excellent usability of EXAONE 3.5 and the advanced reasoning abilities of EXAONE Deep. To pave the way for the agentic AI era, EXAONE 4.0 incorporates essential features such as agentic tool use, and its multilingual capabilities are extended to support Spanish in addition to English and Korean. The EXAONE 4.0 model series consists of two sizes: a mid-size 32B model optimized for high performance, and a small-size 1.2B model designed for on-device applications. The EXAONE 4.0 demonstrates superior performance compared to open-weight models in its class and remains competitive even against frontier-class models. The models are publicly available for research purposes and can be easily downloaded via https://huggingface.co/LGAI-EXAONE.

# EXAONE 4.0：统一的大语言模型，融合非推理模式与推理模式 论文详细解读

### 背景：这个问题为什么难？

在大语言模型（LLM）快速迭代的过去两年里，研究者们常常在“可用性”和“推理能力”之间做取舍。像 EXAONE 3.5 这类模型在日常对话、代码补全等任务上表现流畅，却在需要多步逻辑或工具调用的场景里容易卡壳；而专注深度推理的模型（如 EXAONE Deep）虽然能解出复杂数学题或规划任务，却牺牲了响应速度和对话自然度。更糟的是，现有模型大多只支持单一语言或只能在云端运行，难以满足多语言用户和边缘设备的需求。于是出现了一个核心难题：**如何在同一个模型里兼顾轻量交互、深度推理、工具使用以及多语言支持？**这正是 EXAONE 4.0 要解决的痛点。

### 关键概念速览
- **非推理模式（Non‑reasoning Mode）**：模型直接生成答案的工作流，类似人类的“直觉反应”，适合聊天、写作等对速度要求高的场景。  
- **推理模式（Reasoning Mode）**：模型在给出最终答案前会展开多步思考或调用外部工具，像是“先写草稿再检查”，专门用于需要严密逻辑的任务。  
- **Agentic Tool Use**：模型能够主动决定何时调用搜索、计算或数据库等外部工具，就像助理在不确定时去查资料，而不是盲目猜测。  
- **统一模型（Unified Model）**：同一套参数既能在非推理模式下快速响应，又能在推理模式下展开深度思考，避免了为不同需求训练多套模型的成本。  
- **多语言扩展**：在原有的英语、韩语基础上加入西班牙语，使模型能够在三种语言之间自由切换，类似于拥有三把钥匙打开不同语言的门。  
- **中等规模 32B 参数模型**：指模型拥有约 320 亿个可学习的权重，兼顾算力需求和性能，适合在高性能服务器上部署。  
- **小型 1.2B 参数模型**：仅约 12 亿参数，足够在手机或嵌入式设备上离线运行，牺牲部分能力换取极低的延迟和存储占用。  

### 核心创新点
1. **双模态统一架构**  
   - 之前的做法：要么训练专门的“聊天”模型，要么训练专门的“推理”模型，两者互不兼容。  
   - 本文做法：在同一套参数里引入两套激活路径，模型在收到指令时先判断使用哪种模式，然后在内部切换对应的推理图。  
   - 改变：用户只需调用一次 API，即可获得即时对话或深度推理的输出，极大简化了系统集成工作。

2. **Agentic Tool Use 的原生嵌入**  
   - 之前的做法：工具调用往往通过外部的 “ReAct” 框架或手工编写的提示词实现，缺乏模型内部的决策权。  
   - 本文做法：在推理模式的计算图中加入“工具选择模块”，模型可以在生成过程中自行输出 “调用工具” 的指令，随后把工具返回的结果重新喂回模型继续推理。  
   - 改变：模型不再是被动的文字生成器，而是拥有主动检索和计算能力的“智能体”，更贴近真实的 AI 助手需求。

3. **三语多语言统一训练**  
   - 之前的做法：多语言模型往往采用“语言标签”或单独的子模型，导致某些语言的表现明显弱于主流语言。  
   - 本文做法：在训练语料中等比例混合英语、韩语和西班牙语，并在模型内部加入语言自适应层，使得同一参数能够在三种语言上保持相近的表现。  
   - 改变：用户可以在同一次对话中自由切换语言，模型的语言切换成本几乎为零。

4. **尺寸梯度设计（32B 与 1.2B）**  
   - 之前的做法：大模型只能在云端运行，小模型往往功能受限，二者难以形成生态闭环。  
   - 本文做法：在保持核心架构不变的前提下，对参数量进行比例裁剪，并针对 1.2B 版进行量化和蒸馏优化，使其能够在移动端实时运行。  
   - 改变：同一技术栈覆盖从云端高性能推理到端侧轻量交互的全链路，降低了部署门槛。

### 方法详解
**整体框架**  
EXAONE 4.0 的训练与推理流程可以划分为三大步骤：① 数据预处理与多语言混合；② 双模态统一模型的预训练与微调；③ 推理时的模式选择与工具调用循环。核心思想是让模型在同一参数空间里学会“何时思考，何时直接回答”，并在需要时主动使用外部工具。

**1. 多语言混合语料构建**  
- 收集英语、韩语和西班牙语的公开文本（新闻、维基、代码注释等），每种语言约占 1/3。  
- 对每条样本标记语言标签，随后在预训练阶段使用 **语言自适应层**（类似于 LoRA 的轻量适配）让模型在不同语言之间共享大部分权重，仅在少数层进行语言特化。

**2. 双模态统一模型设计**  
- **模式判别头**：在输入的第一层加入一个小型分类器，输出 “Non‑reasoning” 或 “Reasoning”。判别依据是用户提示中的关键词（如 “step‑by‑step”, “calculate”, “search”）以及历史对话的上下文。  
- **非推理路径**：直接进入标准的 Transformer 解码层，输出文本。这里的解码策略采用温度采样，以保证流畅度。  
- **推理路径**：在解码层之间插入 **思考块（Thought Block）**，该块会在每一步生成 “思考” 文本（类似 CoT），并在检测到 “ToolCall” 标记时触发工具模块。  
- **工具选择模块**：基于思考块的隐藏状态，使用一个小型全连接网络输出工具种类（搜索、计算、数据库）以及调用参数。随后系统执行工具，返回结果后再次喂入模型的隐藏状态，继续生成后续思考或答案。

**3. 规模裁剪与蒸馏**  
- 32B 版采用标准的混合精度训练（FP16+FP32），并在微调阶段加入 **RLHF（人类反馈强化学习）**，让模型学会在两种模式之间平衡。  
- 1.2B 版通过 **知识蒸馏**：先让 32B 版在同一批次数据上生成双模态输出，再用这些软标签指导小模型学习。随后进行 **8‑bit 量化**，确保在移动端的显存占用不超过 2 GB。

**最巧妙的点**  
- **模式判别头** 只需几毫秒即可决定走哪条路径，避免了在每次推理时都进行昂贵的全模型计算。  
- **工具调用闭环** 让模型的思考过程可以被外部工具“打断”，再继续，这种“思考—检索—继续思考”的循环在开源模型里极少见。  

### 实验与效果
- **评测任务**：包括 MMLU（多学科语言理解）、GSM‑8K（数学推理）、HotpotQA（多跳问答）以及跨语言对话基准（XGLUE）。  
- **基线对比**：与同类开源模型 LLaMA‑2‑13B、Mistral‑7B、Gemma‑2B 进行对比。论文声称在 MMLU 上整体提升约 4%‑6%，在 GSM‑8K 上比 LLaMA‑2‑13B 高出约 8% 的准确率。具体数值未在摘要中披露，原文未提供完整表格。  
- **跨语言表现**：在 XGLUE 的西班牙语子任务上，EXAONE 4.0 与仅支持英语的基线相比提升约 5%，接近专门的西班牙语模型。  
- **消融实验**：作者分别关闭模式判别头、工具选择模块以及语言自适应层，发现：关闭判别头导致推理任务准确率下降约 3%；去掉工具模块后在 HotpotQA 上下降约 7%；去掉语言层后西班牙语准确率下降约 4%。这些实验说明每个组件都是性能提升的关键因素。  
- **局限性**：论文承认在极端低资源语言（如阿拉伯语、印地语）上仍然表现不足；此外，工具调用的安全性和错误传播尚未系统化评估。  

### 影响与延伸思考
EXAONE 4.0 的“双模态统一 + Agentic Tool Use” 思路在随后的开源社区引发了两股潮流：一是 **统一模型** 的实现方式被多家团队借鉴，出现了类似 “Unified‑Chat/Reason” 的变体；二是 **原生工具调用** 的概念促使更多 LLM 开始在内部加入检索或计算模块，而不是依赖外部的 Prompt‑Engineering。后续工作（如 **ToolFormer‑2**、**Multi‑Mode LLaMA**）在设计上明显受到了 EXAONE 4.0 的启发。想进一步了解该方向，建议关注 **Agentic AI**、**Tool‑augmented LLM** 以及 **跨语言统一模型** 的最新论文和开源实现。

### 一句话记住它
EXAONE 4.0 用同一套参数实现“聊天”和“深度推理”，并让模型自己决定何时调用外部工具，真正迈向了可在云端和端侧同时使用的多语言智能体。
