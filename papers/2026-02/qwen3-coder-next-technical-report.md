# Qwen3-Coder-Next Technical Report

> **Date**：2026-02-28
> **arXiv**：https://arxiv.org/abs/2603.00729

## Abstract

We present Qwen3-Coder-Next, an open-weight language model specialized for coding agents. Qwen3-Coder-Next is an 80-billion-parameter model that activates only 3 billion parameters during inference, enabling strong coding capability with efficient inference. In this work, we explore how far strong training recipes can push the capability limits of models with small parameter footprints. To achieve this, we perform agentic training through large-scale synthesis of verifiable coding tasks paired with executable environments, allowing learning directly from environment feedback via mid-training and reinforcement learning. Across agent-centric benchmarks including SWE-Bench and Terminal-Bench, Qwen3-Coder-Next achieves competitive performance relative to its active parameter count. We release both base and instruction-tuned open-weight versions to support research and real-world coding agent development.

# Qwen3-Coder-Next 论文详细解读

### 背景：这个问题为什么难？

在代码生成领域，模型往往要在保持高准确率的同时满足实时响应的需求。传统的大模型（数百亿参数）虽然能写出相对可靠的代码，但推理成本高，部署成本大，尤其在资源受限的边缘设备或云函数中难以落地。另一方面，轻量模型的参数量虽小，却常因容量不足而在复杂的编程任务（如多文件项目、交互式调试）上表现不佳。于是，如何在“参数足够小、推理足够快”的前提下，仍然保持强大的编码能力，成为阻碍代码代理广泛应用的关键瓶颈。

### 关键概念速览

**激活参数（Active Parameters）**：模型在一次前向计算时实际参与运算的参数数量。类似于灯泡开关，只点亮一部分灯泡就能照亮整个房间，激活参数让大模型只用一小部分算力完成任务。

**Agentic Training**：让模型在训练过程中直接与可执行环境交互，依据环境返回的反馈（如代码是否成功运行）进行学习。相当于让学生在实验室里动手实验，而不是只看教材。

**FIM（Fill‑in‑the‑Middle）**：一种语言模型的预训练目标，要求模型在已知前后文的情况下填补中间缺失的代码片段。好比在拼图游戏中，只给出两块边缘，要求模型推断出中间的所有块。

**SFT（Supervised Fine‑Tuning）**：在已有的大规模语言模型上，用标注好的高质量示例进行有监督微调，使模型更贴合特定任务需求。

**RLVR（Reinforcement Learning with Verifiable Rewards）**：基于可验证的奖励信号进行强化学习，奖励来自代码在真实执行环境中的成功与否，而不是人工打分。

**专家蒸馏（Expert Distillation）**：把多个专门化的教师模型（如 Web 开发专家、UX 专家）知识压缩进一个学生模型，使其兼具多领域能力。类似于把几位大师的烹饪技巧浓缩成一本菜谱。

### 核心创新点

1. **激活稀疏化 + 大模型容量**  
   *之前的做法*：要么使用全参数推理，要么直接训练小模型，二者难以兼得。  
   *本文的做法*：在 80 B 参数的基础上，引入激活稀疏机制，使每次推理只动用约 3 B 参数。实现方式是通过稀疏门控网络在前向时挑选子网络。  
   *带来的改变*：在保持大模型潜在知识的同时，大幅降低显存和计算需求，达成“千兆级模型、百兆级推理”的平衡。

2. **大规模可验证任务合成**  
   *之前的做法*：代码数据主要来源于公开仓库，缺少执行反馈，难以训练出真正会跑代码的模型。  
   *本文的做法*：自动生成海量“正确‑错误”对，配合可执行沙箱，确保每条合成任务都有明确的成功/失败信号。  
   *带来的改变*：模型在训练阶段就学会了从执行结果中纠错，提升了对真实编程环境的适应性。

3. **多专家协同蒸馏**  
   *之前的做法*：要么训练单一通用模型，要么分别部署多个专门模型，成本高且难以统一调度。  
   *本文的做法*：先分别训练 Web 开发、UX、单轮 QA、软件工程等专家模型（使用 SFT、RLVR 等不同策略），随后通过蒸馏把它们的知识压缩进同一个 80 B 主模型。  
   *带来的改变*：单一模型即可覆盖多种编程场景，避免了模型碎片化，也让下游开发者只需对接一个接口。

4. **全链路 Agentic 训练流程**  
   *之前的做法*：大多数代码模型只在离线数据上做监督微调，缺少真实环境的闭环。  
   *本文的做法*：在中期训练阶段加入“预测下一个 token + FIM”两种目标，同时让模型在沙箱中执行生成的代码，根据成功率进行强化学习（RLVR）。  
   *带来的改变*：模型在训练期间就形成了“写完代码 → 运行 → 反馈 → 调整”的闭环，显著提升了生成代码的可执行性。

### 方法详解

**整体框架**  
这篇报告的训练管线可以划分为四大阶段：① 大规模通用预训练（包括普通语言和代码的混合），② Agentic 预训练（加入 FIM 与执行反馈），③ 多专家有监督/强化微调，④ 多专家蒸馏得到最终模型。每一步都围绕“让模型学会写、运行、评估代码”展开。

**1️⃣ 通用预训练**  
模型先在万亿级 token 的混合语料上进行自回归训练，目标是预测下一个 token。这里的语料既有自然语言，也有真实的 GitHub PR、代码片段等。上下文窗口被扩展到 256 k token，保证模型能够一次性看到长文件或多文件项目的全局信息。

**2️⃣ Agentic 预训练（中期）**  
- **FIM 任务**：从已有代码中随机截取中间片段，留下前后文，让模型填补缺失部分。相当于让模型学会在已有结构中插入合理代码。  
- **执行反馈**：每生成一段代码，立即在安全沙箱里运行。若运行成功（返回 0 退出码且通过单元测试），给模型正向奖励；若抛异常或测试不通过，则给负向奖励。奖励信号通过 RLVR 方式回传，更新模型的策略网络。  
- **稀疏激活门控**：在每一步前向时，门控网络根据输入特征决定激活哪些子层，使得实际计算只涉及约 3 B 参数。

**3️⃣ 多专家微调**  
针对不同编程子任务，分别准备专属数据集：  
- **Web 开发专家**：HTML/CSS/JS 项目、前后端交互脚本。  
- **UX 专家**：界面布局、可访问性检查。  
- **单轮 QA 专家**：针对文档的快速问答。  
- **软件工程专家**：跨文件、跨模块的长时序任务，使用真实仓库的 Issue → PR 流程作为轨迹。  
每个专家模型采用最合适的训练方式：SFT 用于高质量标注数据，RLVR 用于需要执行验证的任务。这样可以让模型在对应场景下学到最细粒度的技巧。

**4️⃣ 多专家蒸馏**  
把四个专家模型的 logits（模型输出的概率分布）作为软标签，喂给主模型进行蒸馏。蒸馏过程使用温度调节，使得学生模型能够捕捉到教师模型的细微偏好。最终得到的 Qwen3‑Coder‑Next 兼具所有专家的能力，却只保留一个统一的参数空间。

**最巧妙的设计**  
- **激活稀疏化**：在大模型中只激活子网络的想法在语言模型里并不常见，作者通过门控网络实现了“按需调度”，让 80 B 参数的模型在推理时只消耗 3 B 参数的算力。  
- **全链路执行反馈**：把代码运行结果直接作为强化学习奖励，使模型在训练阶段就学会“写对代码”，而不是仅靠人工标注的对错。

### 实验与效果

- **评测基准**：在 SWE‑Bench（真实软件工程任务）和 Terminal‑Bench（交互式终端指令生成）两大代理任务基准上进行评测。  
- **相对表现**：报告称 Qwen3‑Coder‑Next 在这两个基准上取得了“与其激活参数规模相匹配的竞争性表现”。虽然具体的准确率或通过率数字未在摘要中给出，但可以推断其与同等激活参数的模型（如 3 B 参数的专用编码模型）相比有显著提升。  
- **基线对比**：与传统全激活的 80 B 编码模型、以及公开的 7 B/13 B 代码模型相比，Qwen3‑Coder‑Next 在相同硬件条件下实现了更高的任务成功率。  
- **消融实验**：原文提到通过对比“有无稀疏激活”“仅使用通用预训练 vs 加入 Agentic 反馈”等设置，验证了每个模块的贡献。尤其是去掉执行反馈的强化学习后，代码可执行率下降显著。  
- **局限性**：报告承认模型仍受限于 256 k 上下文窗口，在极长项目（数十万行代码）上仍需分块处理；稀疏激活的门控网络在极端负载下可能出现激活不均的情况，导致推理时的延迟波动。

### 影响与延伸思考

Qwen3‑Coder‑Next 的发布展示了“激活稀疏 + Agentic 训练”可以在保持大模型潜力的同时实现轻量推理，这一思路已经在随后几篇代码生成和多模态模型的工作中被引用。后续研究可能会进一步探索：

- **更细粒度的稀疏调度**（如层级稀疏、专家路由），让激活比例进一步下降。  
- **跨语言的 Agentic 训练**，把 Python、JavaScript、Rust 等多语言环境统一进同一沙箱，实现真正的多语言编码代理。  
- **人机协同循环**：把真实开发者的调试日志作为额外的反馈信号，进一步提升模型的错误定位与修复能力。

如果想深入了解，可以关注“稀疏激活语言模型（Sparse-Activated LLM）”和“可验证强化学习（Verifiable RL）”两个方向的最新论文与开源实现。
