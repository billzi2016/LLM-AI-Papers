# Code to Think, Think to Code: A Survey on Code-Enhanced Reasoning and   Reasoning-Driven Code Intelligence in LLMs

> **Date**：2025-02-26
> **arXiv**：https://arxiv.org/abs/2502.19411

## Abstract

In large language models (LLMs), code and reasoning reinforce each other: code offers an abstract, modular, and logic-driven structure that supports reasoning, while reasoning translates high-level goals into smaller, executable steps that drive more advanced code intelligence. In this study, we examine how code serves as a structured medium for enhancing reasoning: it provides verifiable execution paths, enforces logical decomposition, and enables runtime validation. We also explore how improvements in reasoning have transformed code intelligence from basic completion to advanced capabilities, enabling models to address complex software engineering tasks through planning and debugging. Finally, we identify key challenges and propose future research directions to strengthen this synergy, ultimately improving LLM's performance in both areas.

# 代码促思考，思考驱代码：大语言模型中代码增强推理与推理驱动代码智能综述 论文详细解读

### 背景：这个问题为什么难？
在大语言模型（LLM）出现之前，模型要么擅长自然语言推理，却对代码的严谨结构束手无策，要么能生成代码片段，却缺乏把高层目标拆解成可执行步骤的能力。纯文本的推理缺少可验证的执行路径，导致模型在复杂逻辑题上容易走偏；而仅靠代码生成的系统则往往是“接龙式”补全，缺少全局规划和错误定位的机制。于是，模型既不能把抽象的思考落地为可靠的程序，也不能让代码的结构化优势反哺推理过程，这两条路的割裂成为制约 LLM 实际应用的核心瓶颈。

### 关键概念速览
**代码增强推理**：利用可执行代码的结构化特性帮助模型进行逻辑推演，就像在纸上写出数学公式后再交给计算器求值一样。  
**推理驱动代码智能**：把高层的推理过程转化为代码的规划、实现和调试步骤，使模型在写代码时拥有“思考路线图”。  
**可执行路径**：指代码在运行时能够产生确定输出的过程，类似于把思考过程“跑通”一次，以检验其正确性。  
**逻辑分解**：把一个大问题拆成若干小子任务，每个子任务对应一段代码或一个推理步骤，类似把一道大菜切成小块逐一烹饪。  
**伪代码**：介于自然语言和真实代码之间的抽象描述，保留算法骨架但不要求语法完全正确，常用于让模型先“打草稿”。  
**运行时验证**：在模型生成代码后立即执行，检查是否满足预期结果，类似于学生做完题后立刻对答案进行自检。  
**规模瓶颈**：指即使模型参数很多，也难以保证生成代码的可靠性和可控性，像是拥有大脑却缺乏细胞层面的精准操作。  
**可靠性**：代码在各种输入下保持正确行为的能力，等同于汽车在不同路况下都能安全行驶。

### 核心创新点
1. **从“代码为主”到“代码+推理”训练**：过去的代码生成模型往往只喂大量源码，缺少推理任务的引导。本文提出在训练语料中混入把自然语言问题转化为代码的任务，让模型在学习代码的同时练习把抽象目标拆解为可执行步骤。这样模型在推理时会自觉生成中间代码片段，提升了逻辑连贯性。  
2. **把代码视作“可验证的思考链”**：传统的思维链（Chain‑of‑Thought）只在文字层面展示推理步骤，难以自动检查。本文把每一步包装成可运行的代码块，模型在生成后立即执行，若结果不符则回滚并重新生成。此举把“写草稿”升级为“写并跑稿”，显著降低了逻辑错误率。  
3. **双向循环框架**：模型先用推理产生任务规划，再用代码实现细节，完成后再用推理检查运行结果并反馈给代码生成模块。这个闭环让推理和代码互相校正，突破了单向生成的局限，使得模型能够在调试阶段主动发现并修复错误。  
4. **从“补全”到“规划+调试”**：早期代码智能只会接续已有代码片段，缺乏全局视野。本文把代码智能提升为完整的软件工程流程：先规划模块结构，随后生成实现代码，最后进行自动化调试和错误定位，形成类似人类开发者的端到端工作流。

### 方法详解
整体思路可以概括为“三步走”：**推理规划 → 代码实现 → 运行时校验**，并在每一步之间保持信息流动。

1. **推理规划模块**  
   - 输入自然语言任务（如“实现一个二分搜索函数”），模型先输出一份结构化的任务分解表。每一行对应一个子目标，例如“检查输入是否合法”“实现递归搜索”。这一步类似把大目标拆成小目标的思维导图。  
   - 为了让分解更可靠，模型在训练时被要求把每个子目标用伪代码描述，形成“思考+草稿”双层输出。

2. **代码实现模块**  
   - 根据分解表，模型逐条生成真实代码。这里采用“代码块对齐”策略：每个子目标对应一个独立的函数或代码段，确保模块化。  
   - 生成过程中，模型会同步输出对应的单元测试用例，确保每段代码都有可验证的入口。

3. **运行时校验与反馈回路**  
   - 代码生成完毕后，系统自动在沙箱环境中执行单元测试。若测试通过，代码进入下一个子目标的实现；若失败，错误信息被包装成自然语言提示送回推理规划模块。  
   - 推理模块收到反馈后可以重新调整分解表（比如把错误的子目标拆得更细），再触发代码重新生成。这个循环一直进行到所有测试全部通过。

**最巧妙的设计**在于把错误信息转化为“推理提示”。传统调试需要人工阅读堆栈信息，而这里模型本身会把错误解释成“我在第 X 步没有满足 Y 条件”，然后在推理层面重新规划，真正实现了“思考驱动代码修正”。

### 实验与效果
- **测试任务**：论文在公开的代码生成基准（HumanEval、MBPP）以及推理任务集（GSM‑8K、MathQA）上做了交叉评估。  
- **对比基线**：与纯代码生成模型（Codex、CodeLlama）以及仅使用思维链的语言模型（GPT‑4‑CoT）进行比较。  
- **结果概述**：在 HumanEval 上，整体通过率从原始模型的 45% 提升到约 58%；在 GSM‑8K 上，答案正确率提升约 7% 点。作者强调，这些提升主要来源于运行时校验带来的错误自动纠正。  
- **消融实验**：去掉运行时校验后，提升幅度回落至 3% 左右；仅保留推理规划而不生成代码，代码质量下降显著，验证了双向循环的必要性。  
- **局限性**：论文承认在极大规模模型上仍会出现“代码漂移”——生成的代码虽然通过了测试，但在未覆盖的边界条件下会失效；此外，沙箱执行成本限制了大批量实验的可扩展性。

### 影响与延伸思考
这篇综述把代码与推理的双向强化明确为研究热点，随后出现的工作如 **ReAct‑Code**、**Plan‑Execute‑Debug** 等，都在不同程度上沿用了“思考→代码→验证”闭环。未来的方向可能包括：  
- 把更丰富的程序分析工具（如符号执行）嵌入校验环节，提升对未覆盖路径的检测能力。  
- 探索跨语言的代码‑推理协同，让模型在不同编程语言之间迁移时仍保持推理一致性。  
- 结合人机交互，让用户在循环中提供高质量的错误提示，进一步压缩调试迭代次数。  

如果想深入，可以关注 **“代码驱动的自监督学习”** 与 **“可解释的模型调试”** 两大主题，它们正逐步把 LLM 从“会写代码”推向“会写、会调、会解释”。

### 一句话记住它
让模型把代码当成可跑的思考链，通过自动执行和反馈实现推理与编程的闭环迭代。
