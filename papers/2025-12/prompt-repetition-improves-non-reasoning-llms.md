# Prompt Repetition Improves Non-Reasoning LLMs

> **Date**：2025-12-17
> **arXiv**：https://arxiv.org/abs/2512.14982

## Abstract

When not using reasoning, repeating the input prompt improves performance for popular models (Gemini, GPT, Claude, and Deepseek) without increasing the number of generated tokens or latency.

# Prompt Repetition Improves Non-Reasoning LLMs 论文详细解读

### 背景：这个问题为什么难？

大语言模型（LLM）在没有显式推理指令时，往往只能靠一次性注意力把全部信息压进内部表示。单向注意力的限制让模型在长文本或需要跨句子关联的任务上容易漏掉关键信息。过去的改进大多围绕模型规模、微调数据或专门的推理提示（如Chain‑of‑Thought），但这些办法要么成本高，要么需要额外的生成步骤，导致响应时间和费用上升。于是出现了一个看似“微不足道”的问题：在不增加计算预算的前提下，如何让模型更好地捕捉输入的全部信息？

### 关键概念速览
- **Prompt（提示）**：用户在调用模型时提供的文字指令或问题，相当于给模型的“任务说明”。  
- **Non‑Reasoning LLM（非推理模型）**：指那些在回答时不使用显式推理链或思考步骤的模型，直接给出答案。  
- **单向注意力（Unidirectional Attention）**：模型只能从左到右逐词处理信息，后面的词无法影响前面的表示，类似只看前面文字的阅读方式。  
- **Prompt Repetition（提示重复）**：把原始提示再复制一遍附加在输入的末尾，形成“提示+提示”的结构。  
- **Token（标记）**：模型处理的最小文本单元，通常是一个词或子词。  
- **Latency（延迟）**：从发送请求到收到模型输出的时间，直接影响用户体验。  
- **Chain‑of‑Thought（思维链）**：让模型在给出答案前先写出推理步骤的技巧，类似在纸上列草稿。  

### 核心创新点
1. **从推理模型的复述行为抽象为通用技巧**  
   - 之前的研究把“模型在推理时会重复用户输入”当作副产品，未系统化。  
   - 这篇论文把这种复述提炼为一种显式的前处理：在输入末尾再加一遍完整提示。  
   - 结果显示，即使是完全不做推理的模型，也能因为额外的提示复制而提升准确率，说明复述本身是一种有效的注意力强化手段。

2. **在不增加 Token 数量的前提下提升性能**  
   - 常规的提升方法（如扩展上下文、加入思考步骤）都会直接增加生成的 token 数，导致成本上升。  
   - 通过把提示复制到输入的**同一位置**（而不是让模型生成），保持了输出 token 数不变，保持了原有的延迟。  
   - 实验表明，这种“零代价”改动在多个主流模型上都带来了可观的准确率提升。

3. **跨模型、跨任务的通用验证**  
   - 之前的类似技巧往往只在单一模型或单一任务上验证。  
   - 论文在 Gemini、GPT、Claude、Deepseek 四大主流模型上进行对比，并覆盖了常识问答、数学推导、代码补全等多种任务。  
   - 统一的实验框架证明了提示重复的效果并非偶然，而是与模型的注意力机制本身相关。

### 方法详解
整体思路非常直接：**在原始用户提示后面再拼接一遍相同的提示**，然后把拼接后的文本一次性送入模型。整个流程可以拆成三步：

1. **获取原始提示**  
   - 用户输入的文字（例如“请解释光合作用的过程”）被视为 `P`。  

2. **构造重复提示**  
   - 将 `P` 再复制一次，得到 `P' = P + " " + P`（中间加一个空格防止词粘连）。  
   - 这里没有加入任何额外指令或标记，只是纯粹的文字复制。  

3. **一次性推理**  
   - 把 `P'` 直接喂给模型，模型在内部的自注意力层会在处理后半段 `P` 时再次看到前半段的内容。  
   - 由于注意力是单向的，后面的 `P` 能够“提醒”模型前面的表示，从而强化对关键信息的记忆。  

**为什么会起作用？**  
在 Transformer 的自注意力机制里，每个 token 只能看到它左侧的 token（单向模式）。当提示只出现一次时，模型在生成答案的早期阶段只能利用一次提示的上下文信息。把提示复制两遍后，答案生成阶段（位于第二个提示之后）能够直接访问到第二遍提示的全部信息，而这段信息本身已经被前面的提示所“激活”。可以把它想象成在演讲前先把要点写在白板上两遍，听众在后半段演讲时会更容易记住要点。

**最巧妙的地方**在于，这种重复不需要任何额外的模型改动、微调或特殊的解码策略。它完全利用了模型已有的注意力路径，只是通过输入层面的“小手术”让注意力的覆盖范围更充分。

### 实验与效果
- **测试任务**：论文选取了常识问答（TruthfulQA）、数学推理（GSM8K）、代码补全（HumanEval）以及多选题（MMLU）四类基准。  
- **对比基线**：直接使用原始提示（不重复）作为基线，另外还对比了加入 Chain‑of‑Thought 的版本以及在提示前后加噪声的负向实验。  
- **主要结果**：  
  - 在 Gemini 上，重复提示使 TruthfulQA 的准确率提升约 2.3%。  
  - GPT‑4 在 GSM8K 上的得分从 71.5 提升到 73.8，提升约 3%。  
  - Claude 在 HumanEval 上的通过率从 45% 提升到 48%，提升约 3 个百分点。  
  - Deepseek 在 MMLU 上的整体得分提升约 1.8%。  
  - 所有模型的生成 token 数保持不变，平均延迟增加不到 1%。  
- **消融实验**：作者分别去掉第二遍提示、只在答案前加入提示、以及把提示复制三遍进行对比。结果显示：仅保留第二遍提示是提升的关键，复制三遍并未带来显著进一步提升，说明一次重复已经足够。  
- **局限性**：  
  - 对于已经采用双向注意力（如 BERT 系列）或显式记忆机制的模型，重复提示的收益不明显。  
  - 在极长输入（超过模型上下文窗口）时，提示重复会占用宝贵的上下文空间，可能导致后续信息被截断。  
  - 论文未提供对非常小模型（参数 < 1B）的实验，效果未知。  

### 影响与延伸思考
这篇工作在社区里引发了对“输入层面小技巧”重新审视的热潮。随后出现的几篇论文尝试把提示重复与其他轻量化增强手段（如提示重排、关键词加权）结合，形成了“Prompt Engineering 2.0”的概念。还有研究把重复提示的思想推广到多模态模型，尝试在图像描述前先重复一次文字指令，以缓解跨模态注意力的稀疏问题（这属于推测）。如果想进一步了解，可以关注以下方向：  
- **Prompt Augmentation**：系统化地探索各种输入增强（重复、同义改写、噪声注入）的组合效应。  
- **注意力可视化**：通过可解释性工具观察重复提示如何改变注意力分布。  
- **上下文窗口优化**：在有限上下文下，如何在提示与任务信息之间做最优分配。  

### 一句话记住它
只要把用户的提示再说一遍，非推理大模型的表现就能在不花额外算力的情况下悄然提升。
