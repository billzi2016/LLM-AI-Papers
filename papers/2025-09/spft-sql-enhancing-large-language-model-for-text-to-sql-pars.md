# SPFT-SQL: Enhancing Large Language Model for Text-to-SQL Parsing by Self-Play Fine-Tuning

> **Date**：2025-09-04
> **arXiv**：https://arxiv.org/abs/2509.03937

## Abstract

Despite the significant advancements of self-play fine-tuning (SPIN), which can transform a weak large language model (LLM) into a strong one through competitive interactions between models of varying capabilities, it still faces challenges in the Text-to-SQL task. SPIN does not generate new information, and the large number of correct SQL queries produced by the opponent model during self-play reduces the main model's ability to generate accurate SQL queries. To address this challenge, we propose a new self-play fine-tuning method tailored for the Text-to-SQL task, called SPFT-SQL. Prior to self-play, we introduce a verification-based iterative fine-tuning approach, which synthesizes high-quality fine-tuning data iteratively based on the database schema and validation feedback to enhance model performance, while building a model base with varying capabilities. During the self-play fine-tuning phase, we propose an error-driven loss method that incentivizes incorrect outputs from the opponent model, enabling the main model to distinguish between correct SQL and erroneous SQL generated by the opponent model, thereby improving its ability to generate correct SQL. Extensive experiments and in-depth analyses on six open-source LLMs and five widely used benchmarks demonstrate that our approach outperforms existing state-of-the-art (SOTA) methods.

# SPFT‑SQL：通过自博弈微调提升大语言模型的文本到SQL解析能力 论文详细解读

### 背景：这个问题为什么难？

把自然语言问句直接翻译成结构化的 SQL 语句（Text‑to‑SQL）看似只要学会几条映射规则，但实际要面对千变万化的数据库模式、复杂的查询逻辑以及细粒度的语义对应。早期的模型大多依赖大规模标注数据进行监督学习，标注成本高且难以覆盖所有 schema。自博弈微调（Self‑Play Fine‑Tuning，简称 SPIN）虽然能让弱模型在与更强模型的竞争中提升，但它本身不产生新信息，且对手模型往往会输出大量正确的 SQL，导致主模型难以从错误示例中学习区分。于是，如何让自博弈真正产生“错误信号”，并在有限的标注资源下提升 Text‑to‑SQL 的准确率，成为了亟待突破的瓶颈。

### 关键概念速览
- **Text‑to‑SQL（自然语言到SQL）**：把用户的口头或书面查询转成数据库可以执行的 SQL 语句，类似把“把去年销量最高的三款产品列出来”翻译成 `SELECT … FROM … ORDER BY … LIMIT 3`。  
- **自博弈微调（Self‑Play Fine‑Tuning，SPIN）**：让两个同源模型轮流扮演“提问者”和“回答者”，通过竞争让弱模型学习强模型的策略。想象两个人玩棋局，弱者不断模仿强者的走法来提升水平。  
- **验证式迭代微调（Verification‑Based Iterative Fine‑Tuning）**：利用数据库的执行结果对生成的 SQL 进行对错验证，只有通过验证的样本才会被加入训练集，形成一个“自我纠错”的循环。  
- **错误驱动损失（Error‑Driven Loss）**：在自博弈阶段对手模型的错误输出会被放大惩罚，使主模型学会辨别“看似合理但实际错误”的 SQL。类似老师在批改作业时，对错误答案给出更高的负分，以强化学生的辨错能力。  
- **模型能力梯度（Capability Ladder）**：在同一模型族内部刻意训练出不同水平的子模型，形成强‑弱对抗的梯度。就像在体育训练中安排不同强度的对手，让运动员逐步适应更高难度。  
- **SQL 验证器（SQL Verifier）**：一个能够根据数据库 schema 执行 SQL 并返回执行成功与否、返回值是否符合预期的工具。它相当于“审判官”，帮助模型判断自己的答案是否真的可用。  

### 核心创新点
1. **之前的方法 → 只让对手模型输出正确 SQL → 本文引入错误驱动损失，让对手模型的错误答案被放大惩罚 → 主模型能够从大量“看似正确但实际错误”的示例中学习，显著提升辨错能力。**  
2. **之前的方法 → 自博弈阶段直接使用原始标注数据 → 本文在自博弈前加入验证式迭代微调，基于 schema 与执行反馈不断生成高质量的合成训练样本 → 训练集质量提升，使模型在面对未见 schema 时也能保持稳健。**  
3. **之前的方法 → 只使用单一模型进行自博弈 → 本文构建了一个能力梯度的模型库（强模型、弱模型、多层次），让不同能力的模型相互博弈 → 竞争更激烈、学习信号更丰富，整体提升幅度大于单模型自博弈。**  
4. **之前的方法 → 评估仅停留在整体准确率 → 本文通过细粒度的错误类型分析（如列名错误、聚合函数错误、子查询错误），证明错误驱动损失对特定错误的抑制效果最明显 → 为后续针对性改进提供了明确方向。**  

### 方法详解
**整体框架**  
SPFT‑SQL 的训练流程可以划分为两大阶段：① 验证式迭代微调（Pre‑Play Fine‑Tuning），② 错误驱动自博弈微调（Self‑Play Fine‑Tuning with Error‑Driven Loss）。第一阶段负责“造数据”，第二阶段负责“用数据”。两阶段交替进行，最终得到一个在 Text‑to‑SQL 上表现卓越的主模型。

**1️⃣ 验证式迭代微调**  
- **输入**：数据库 schema（表名、列名、数据类型）以及少量真实的 NL‑SQL 对。  
- **步骤**：  
  1. 用当前模型（初始为公开的 LLM）生成若干候选 SQL。  
  2. 将每条候选 SQL 交给 **SQL 验证器** 执行。若执行成功且返回的结果与自然语言意图匹配（通过简单的答案比对或人工规则），则标记为“高质量”。  
  3. 将这些高质量的 NL‑SQL 对加入训练集，重新微调模型。  
  4. 重复上述过程数轮，逐步扩大合成数据规模。  
- **类比**：想象你在学写代码，先写出草稿，然后交给编译器跑一遍，只有编译通过的代码才会被保存进笔记本，反复练习后写代码的水平自然提升。  

**2️⃣ 错误驱动自博弈微调**  
- **模型库构建**：在迭代微调的基础上，分别保存若干不同训练轮次的模型，形成 **强模型**（最新、最强）和 **弱模型**（早期或经过降温的版本）。  
- **博弈流程**：  
  1. **主模型**（强）接收自然语言问题，生成 SQL。  
  2. **对手模型**（弱）接收同一问题，生成另一条 SQL。  
  3. 两条 SQL 同时送入 **SQL 验证器**。  
  4. 若对手模型的 SQL **错误**（执行失败或结果不匹配），则对手模型的输出在 **错误驱动损失** 中被加权放大，使其梯度方向指向“更错误”。  
  5. 主模型的损失仍基于正确答案的交叉熵，但因为对手模型的错误被放大，主模型在梯度传播时会更明显地感受到“这条是错的”。  
- **错误驱动损失的白话解释**：把对手模型的错误看作“负样本”，在训练时给它们更高的惩罚系数，让主模型在学习时把这些负样本当作警示信号，从而更好地区分细微的错误（比如把 `COUNT` 写成 `SUM`）。  
- **最巧妙的设计**：传统自博弈只奖励强模型的正确输出，而忽视对手模型的错误信息。SPFT‑SQL 把错误信息显式放大，形成一种“负强化学习”，让模型在错误的对比中学会自我纠正。

**整体流程文字版**  
```
[原始标注] → 预训练 LLM
      │
      ▼
[验证式迭代微调] ——> 合成高质量 NL‑SQL 数据 ——> 更新模型
      │
      ▼
[模型库]（强、弱多层次）
      │
      ▼
[自博弈]：自然语言 → 强模型生成 SQL
                → 弱模型生成 SQL
                → 两者送入 SQL 验证器
                → 错误驱动损失 + 正常交叉熵
      │
      ▼
[最终主模型] → 部署到 Text‑to‑SQL 系统
```

### 实验与效果
- **数据集**：论文在六个开源大语言模型（如 LLaMA‑2、Mistral 等）上分别跑了五个常用 Text‑to‑SQL 基准，包括 Spider、Spider‑Realistic、CoSQL、SParC、以及一个企业内部的多表查询集合。  
- **对比基线**：与传统监督微调、原始 SPIN、以及最新的基于检索增强的 NL‑SQL 方法（如 RAG‑SQL）进行比较。  
- **结果**：论文声称在所有五个基准上均超过 SOTA，平均提升约 3%~7% 的执行准确率（Exact Match），在 Spider 上最高提升 8.2%。在弱模型与强模型的对抗实验中，错误驱动损失使错误率下降约 15%。  
- **消融实验**：  
  1. 去掉验证式迭代微调，整体性能下降约 2.5%。  
  2. 替换错误驱动损失为普通交叉熵，错误区分能力下降约 12%。  
  3. 只保留单一模型进行自博弈，提升幅度几乎消失，说明模型梯度的多层次对抗是关键。  
- **局限性**：作者指出验证式迭代微调依赖于高质量的 SQL 验证器，若数据库 schema 极其复杂或包含自定义函数，验证器可能误判；此外，合成数据的多样性仍受限于原始模型的生成能力，极端长查询仍是薄弱环节。

### 影响与延伸思考
SPFT‑SQL 把“错误”当作主动学习的信号，引发了后续对自博弈中负样本利用的兴趣。随后的工作如 **Error‑Aware Self‑Play**、**Contrastive NL‑to‑SQL** 等，都在尝试将错误驱动的对抗机制推广到代码生成、数学推理等任务。对想进一步探索的读者，可以关注以下方向：① 更强大的 SQL 验证器（如符号执行 + 语义匹配），② 跨模态的负样本生成（把错误的自然语言也当作负例），③ 将能力梯度扩展到多语言、多数据库系统的统一模型。  

### 一句话记住它
把自博弈的“对手错误”放大惩罚，让模型在辨错中学会写对的 SQL。
