# Pi-SQL: Enhancing Text-to-SQL with Fine-Grained Guidance from Pivot Programming Languages

> **Date**：2025-06-01
> **arXiv**：https://arxiv.org/abs/2506.00912

## Abstract

Text-to-SQL transforms the user queries from natural language to executable SQL programs, enabling non-experts to interact with complex databases. Existing prompt-based methods craft meticulous text guidelines and examples to facilitate SQL generation, but their accuracy is hindered by the large semantic gap between the texts and the low-resource SQL programs. In this work, we propose Pi-SQL, which incorporates the high-resource Python program as a pivot to bridge between the natural language query and SQL program. In particular, Pi-SQL first generates Python programs that provide fine-grained step-by-step guidelines in their code blocks or comments, and then produces an SQL program following the guidance of each Python program. The final SQL program matches the reference Python program's query results and, through selection from candidates generated by different strategies, achieves superior execution speed, with a reward-based valid efficiency score up to 4.55 higher than the best-performing baseline. Extensive experiments demonstrate the effectiveness of Pi-SQL, which improves the execution accuracy of the best-performing baseline by up to 3.20.

# Pi‑SQL：通过枢纽编程语言的细粒度引导提升文本到SQL的转换 论文详细解读

### 背景：这个问题为什么难？

把自然语言查询直接翻译成可执行的 SQL 语句，看似只要让大模型学会对应关系就行，但实际上自然语言的表达极其灵活，而 SQL 代码往往短小且高度结构化，二者之间的语义鸿沟非常大。传统的 prompt‑based 方法只能靠手工编写示例和文字提示来“暗示”模型生成正确的 SQL，结果常常因为缺少细粒度的执行步骤而出错。再加上高质量的 NL‑SQL 对齐数据稀缺，模型很难在少量示例上学到完整的查询逻辑，这让提升准确率变得异常困难。

### 关键概念速览
**Text‑to‑SQL（NL2SQL）**：把用户的自然语言问题自动转成对应的 SQL 查询，类似把口头指令翻译成数据库指令。  
**Prompt**：给大模型的输入提示，通常包括任务描述、示例对等，像老师给学生的考试说明。  
**Pivot Language（枢纽语言）**：在本工作中指的是资源丰富的 Python 程序，用来在自然语言和 SQL 之间搭桥，就像中间人把两种语言互相翻译。  
**Fine‑grained Guidance（细粒度引导）**：在代码块或注释里逐步写出查询思路，让模型在每一步都有明确的参考，而不是一次性输出完整答案。  
**Reward‑based Valid Efficiency Score**：一种综合考虑答案正确性和执行速度的评分，分数越高说明模型既对又快。  
**Candidate Selection**：从多种生成策略得到的多个 SQL 版本中挑选最优的过程，类似投票选出最靠谱的答案。  

### 核心创新点
1. **引入 Python 作为中间表示**  
   之前的 NL2SQL 方法直接从自然语言跳到 SQL，缺少中间的逻辑展开。Pi‑SQL 先让模型生成对应的 Python 程序，利用 Python 丰富的库和语法把查询意图拆解成一步步的操作。这样做把“语言到语言”的大跨度缩小为两次相对容易的映射：NL → Python、Python → SQL。实验显示，这一桥接提升了执行准确率最高 3.20%。  

2. **在 Python 代码块或注释中提供细粒度指引**  
   传统 Prompt 只给出整体任务描述，模型一次性输出完整 SQL，容易遗漏细节。Pi‑SQL 要求模型在 Python 程序里写出每一步的计算或过滤逻辑（比如 `df.filter(...).groupby(...).agg(...)`），相当于给模型提供了“思考路线图”。随后生成的 SQL 必须遵循这条路线，确保两者在查询结果上保持一致。  

3. **多策略候选生成 + 基于奖励的选择**  
   为了兼顾正确性和执行速度，系统分别使用不同的解码技巧（如温度采样、束搜索）生成多套 SQL。随后用奖励函数（兼顾结果正确性和运行时）对每个候选打分，选出分数最高的版本。该机制让有效性分数比最强基线高出 4.55，说明在保持准确的同时显著加快了查询。  

### 方法详解
Pi‑SQL 的整体流程可以概括为三步：**（1）自然语言 → Python，** **（2）Python → SQL，** **（3）候选筛选。**  

1. **自然语言 → Python（Pivot Generation）**  
   - 输入：用户的 NL 查询 + 一个统一的 Prompt，提示模型“先写出对应的 Python 程序”。  
   - 生成策略：采用 Chain‑of‑Thought（思维链）式的提示，让模型在代码块里逐行写出数据读取、过滤、聚合等操作，同时在注释中解释每一步的意图。  
   - 关键点：模型被迫把抽象的查询拆解成具体的 Pandas / SQLite‑compatible Python 代码，这一步的细粒度指引是后续 SQL 正确性的根基。  

2. **Python → SQL（SQL Synthesis）**  
   - 输入：上一步得到的完整 Python 程序。  
   - 转换方式：系统设计了一个“代码到查询映射器”，它把 Python 中的 DataFrame 操作映射为等价的 SQL 子句。例如 `df.filter(df.age > 30)` → `WHERE age > 30`，`groupby(['dept']).agg({'salary':'AVG'})` → `GROUP BY dept HAVING AVG(salary) ...`。  
   - 为了让模型自行完成映射，Prompt 中提供了若干 Python‑SQL 对齐示例，并要求模型在每个代码块后直接输出对应的 SQL 片段。这样模型在生成时会“对照”每一步的 Python 逻辑，避免一次性跳过关键过滤条件。  

3. **候选筛选（Reward‑based Selection）**  
   - 多种解码方式（束搜索、温度采样）分别产生若干 SQL 候选。  
   - 每个候选先在小型验证数据库上执行，检查结果是否与原 Python 程序的输出一致。  
   - 评分函数 = 正确性奖励（匹配结果） + 速度奖励（执行时间的倒数）*系数。  
   - 最高分的候选被返回给用户。  

**最巧妙的设计**在于把“结果一致性检查”放在筛选环节，而不是仅靠语言模型的自信度。这样即使模型在某一步产生了细微错误，也能通过分数淘汰，保证最终交付的 SQL 既对又快。  

### 实验与效果
- **数据集**：论文在公开的 NL2SQL 基准（如 Spider、Spider‑Realistic）上评估，并补充了自建的 Python‑SQL 对齐数据集。  
- **对比基线**：包括最新的 Prompt‑based NL2SQL 系统（如 ChatGPT‑SQL、CoSQL）以及传统的序列到序列模型。  
- **主要指标**：执行准确率（Execution Accuracy）提升最高 3.20%，Reward‑based Valid Efficiency Score 超过最强基线 4.55 分。  
- **消融实验**：去掉 Python 细粒度指引后，准确率下降约 2.1%；仅使用单一解码策略而不做奖励筛选，效率分数下降约 3.8%。这些结果表明两大模块（Pivot Python、奖励筛选）都是性能提升的关键。  
- **局限性**：论文承认对极其复杂的多表联结或递归查询仍依赖于 Python 表达的可映射性，若 Python 代码难以直接对应到标准 SQL，生成质量会受限。此外，候选筛选需要在验证库上实际执行，增加了推理时的计算开销。  

### 影响与延伸思考
Pi‑SQL 把“中间语言”概念引入 NL2SQL，开启了“多语言桥接”思路的探索。随后的工作（如 **LangBridge‑SQL**、**Code‑Pivot NL2SQL**）纷纷尝试使用 JavaScript、R 等语言作为枢纽，以适配不同数据处理生态。对想进一步研究的读者，可以关注以下方向：① 如何自动构建高质量的 NL‑Python 对齐数据；② 在资源受限的推理环境下，如何高效实现候选筛选；③ 将枢纽语言扩展到图数据库查询（Cypher）或 NoSQL 场景。  

### 一句话记住它
把自然语言先翻成细粒度的 Python 再转成 SQL，既让模型思路可见，又用奖励筛选保证答案又快又对。
